# AGENTS.md — toppers-skills リポジトリ運用契約

> このリポジトリは **TOPPERS系RTOS（ITRON系）開発のための「汎用・実装非依存」skill** を
> 集約・キュレーションする場である。全AIコーディングツール（Claude Code / Cline / Cursor 等）
> 共通の正本。**本ファイルが分担・命名・形式・提案フローの唯一の正本**。
>
> 配下の各リポジトリ（asp3_core / asp3_tz_work / 各SDK統合リポ 等）のClaudeが、自リポから
> skill化すべき情報を見つけたとき、**何をどこに置くか**で迷わないための契約。

---

## 1. このリポジトリの役割（と、役割でないこと）

| ✅ ここに置く（汎用・実装非依存） | ❌ ここに置かない（実装固有） |
|---|---|
| ITRON/μITRON 仕様、TOPPERS共通の規約・概念 | 特定リポのビルドコマンド・プリセット名 |
| 静的生成・禁則・命名・エラー意味論 | 特定ツール名（スクリプト・ドライバ）・ファイルパス |
| 移植・上流追従・診断の**考え方** | 特定ターゲット名（mps2_an505 等）・出力形式 |
| 複数のTOPPERS派生で再利用できる知識 | あるリポだけで通用する手順・台帳ファイル名 |

> **判定の一言**：「別のTOPPERS派生でもそのまま通じるか？」
> 通じる＝汎用＝ここ。通じない（コマンド・パス・固有名に依存）＝実装固有＝**そのリポの
> `.claude/skills/` に置く**。

---

## 2. 汎用 / 実装固有 の分担（最重要）

### 2.1 汎用skill ＝ このリポジトリ

実装非依存の知識のみ。具体コマンド・固有名・出力形式・ファイル名を**書かない**。
必要なら「具体手順はリポジトリ固有（各リポの `AGENTS.md`/`docs`/専用skill）」と委譲する。

### 2.2 実装固有skill ＝ 各開発リポジトリの `.claude/skills/`

ビルド/実行/解析の具体コマンド、プリセット、スクリプト、ターゲット名、台帳ファイル名、
出力形式などは、**そのリポジトリの `.claude/skills/<name>/` に置く**。汎用skillへ
リンクして概念を委譲し、自分は「このリポでの叩き方」に徹する。

> 例（asp3_core）：固有は `.claude/skills/asp3-core-ops/`、概念は本リポの
> `toppers-kernel-dev`/`toppers-kernel-debug`/`toppers-asp` を参照。
> `.claude/` を gitignore しているリポは `!.claude/skills/` で skills だけ追跡対象にする。

## 3. 現在の汎用skill 一覧

| skill | 対象 | 概要 |
|---|---|---|
| `toppers-asp` | **アプリ開発** | タスク・セマフォ・イベントフラグ・データキュー・ミューテックス・固定長メモリプール等の Cコード／`.cfg` の書き方。API/静的API/エラー辞書＋例示。μITRON4.0との差分。 |
| `toppers-kernel-dev` | **カーネル側 作業規約** | カーネル/アーキ/ターゲット側を扱う共通規約：二大禁則・静的生成・層構造・移植・上流追従マージ。 |
| `toppers-kernel-debug` | **診断・検証** | ハードフォルト/ハング/優先度逆転/コンテキストエラー等の症状→原因、実行の観測、TTSP3の概念。 |

役割の境界：
- アプリの**書き方** → `toppers-asp`
- カーネル側の**やってよい/いけない・作業順序** → `toppers-kernel-dev`
- 動かして**おかしいときの切り分け** → `toppers-kernel-debug`

新しい汎用skillを足すときは、この3つと重複しないか・どれかの references で足りないかを先に確認する。

## 4. 命名規約

| 種別 | 接頭辞 | 例 |
|---|---|---|
| 汎用（このリポ） | `toppers-*` | `toppers-asp` / `toppers-kernel-dev` / `toppers-kernel-debug` |
| 実装固有（各リポ） | `<repo>-ops` 等 | `asp3-core-ops` |

- 対になる汎用skillは並列命名で揃える（`toppers-kernel-dev` ↔ `toppers-kernel-debug`）。
- 機能が別なら統合せず別skillにする（規約と診断は混ぜない）。

## 5. skill のファイル形式・レイアウト

```
toppers-skills/
  <skill-name>/
    skill/
      SKILL.md            ← 必須。frontmatter + 本体
      references/*.md      ← 任意。詳細リファレンス（SKILL.md から参照）
      examples/...         ← 任意。サンプル（toppers-asp 方式）
```

`SKILL.md` の frontmatter（必須）：
```yaml
---
name: <skill-name>            # ディレクトリ名と一致
description: <発動条件を具体語で。日本語＋英語キーワード。
             「ユーザが〜と言ったとき」「〜を扱うとき」を列挙。
             他skillとの境界（どれを使い分けるか）も明記する>
---
```

本体の作法：
- 冒頭で**他skillとの役割分担**を明示（誤発動・重複を防ぐ）。
- 概念は本文、詳細は `references/` に分割（SKILL.md は肥大化させない）。
- **実装固有の語を入れない**（§2）。具体手順は「リポジトリ固有」と委譲する。

## 6. 各リポのClaudeへの委譲フロー（提案 → 統合）

自リポで「skill化すべき情報」を見つけたら：

```
1. 仕分ける
   ├─ 実装固有（コマンド/パス/固有名/出力形式に依存）
   │     → 自リポの .claude/skills/<repo>-ops/ に直接書く（自分の所有物）
   └─ 汎用（別のTOPPERS派生でも通じる概念）
         → このリポへの「提案」にする（直 push しない）

2. 汎用候補の提案方法（どちらか）
   ├─ ブランチ＋PR：feat/<topic> で SKILL.md/references を追加し PR
   └─ 提案メモ：既存skillのどこに何を足すかを Issue/メモで上げる
      （重複・命名・矛盾はキュレーターが判断）

3. キュレーター（リポ管理者 or 指定session）が
   - 既存3 skill との重複を排除（dedup）
   - 命名規約・形式・実装非依存性（§2/§4/§5）を点検
   - 取り込む or 差し戻す
```

**禁止**：各リポのClaudeが汎用skillに実装固有の語を混ぜて直 push すること。
迷ったら「別リポでも通じるか？」で判定し、通じないなら自リポへ。

> 各開発リポで棚卸しさせるための**定型プロンプト**は [`docs/harvest-prompt.md`](docs/harvest-prompt.md)。
> そのリポのClaudeに貼れば、固有skillを自リポに作り、汎用候補を提案として上げてくれる。

## 7. 上流ソース系ディレクトリの扱い

`TTSP3`・`FMP3`・`HRP3` 等の**上流取り込み元は改変しない**＝skillの“ソースとして”は
扱うが、それらリポ自体に skill を置かない。そこから得た**汎用知識**（例：TTSP3の概念、
FMP3＝マルチプロセッサ／HRP3＝メモリ保護といった派生の違い）は、本リポの汎用skillへ
提案として反映してよい。作業用ディレクトリ（一時的なもの）の知見は、固まったら
親となる開発リポの skill/docs に集約する。

## 9. 品質の鉄則

- **実装非依存性を守る**：汎用skillに固有コマンド・パス・ターゲット名・出力形式を書かない
  （`grep` で固有語が混ざっていないか点検する）。
- **裏取り**：固有skillの手順は、そのリポで実際にビルド/実行して検証してから書く。
- **重複させない**：同じ概念を複数skillに書かない。references で一本化し相互リンク。
- **発動の明確化**：description はトリガー語を具体的に。他skillとの使い分けを必ず書く。
