# 実行時間の計測（性能観測）— 実装非依存

コンテキストスイッチやサービスコールの所要時間を、実行時間分布(histogram)サービスや
時刻取得で測るときの、TOPPERS系で共通の作法と、エミュレータ上の落とし穴。

> 具体のフラグ名・レジスタ番地・ビルドコマンド・ターゲット名はリポジトリ固有
> （各リポの専用skill/`docs`）。ここは概念のみ。

## 1. 計測の時間源は差し替えられる（層構造の定石）

TOPPERS の実行時間分布計測サービスは、計測の**時間源を差し替え可能**な作りになっている。
既定の時間源は低分解能のシステム時刻（マイクロ秒級）だが、コンテキストスイッチや
サービスコールはサブマイクロ秒なので、この分解能では量子化されてしまう。サイクル/
ナノ秒で測りたいときは、アーキのサイクルカウンタを時間源にする。差し替えはアーキ横断で
同じ層構造で行うのが定石：

- **サービス本体は無改変**。サービスが持つ「時間源マクロが未定義なら既定」というフックを、
  アーキ共通層が上書きする（時刻のデータ型・現在値取得）。
- **オプトインフラグで有効化**する（既定ビルドは無影響＝低分解能のまま）。常時有効化は
  デバッグ/トレース系ユニットの電力・副作用を生むので避ける。
- **カウンタの有効化はコア初期化**で、同じオプトインフラグの下で行う。
- **生カウント→時間単位への変換はクロック依存**。変換係数はボード（クロック周波数）に
  依存するので、**ターゲット/ボード層**が与える（アーキ共通層は生カウントのまま返す）。

新アーキに足すときは、**他アーキの同じフック（アーキ共通の syssvc コア依存ヘッダ等）の
有無を先に確認し、同型に倣う**。新しいフラグ/ヘッダを発明しない。

*〔分類: 観測/性能計測（＋制約/アーキ構造）｜ プロセッサ: 非依存（カウンタ実体の例＝
ARM Cortex-A: サイクル PMU／Cortex-M: DWT サイクルカウンタ／RISC-V: cycle CSR）｜
出所: asp3_core〕*

## 2. エミュレータは性能/サイクルカウンタを実装しないことが多い

サイクルカウンタや性能モニタは、エミュレータ（命令エミュレーションは行うがサイクル精度
ではない実行環境）では**未実装で 0／一定**を返すことが多い。さらに厄介なのは、アーキ的に
「カウンタ非実装」を示す**状態ビットが 0（＝実装あり）のまま**で、レジスタ越しには実装
されているように見えること（書き込みも受け付けるが計数しない）。

- **状態ビットを信用しない**。計測前に「既知の空ループ前後でカウンタが実際に進むか」を
  **実行時プローブ**し、進まなければその実行環境は非対応と判断する。
- 非対応なら**低分解能の時刻源へフォールバック**するか、テストを SKIP し、**実機でのみ**
  サイクル/ナノ秒計測を行う。
- 「実装ありに見えるのに 0」という症状を、コード側のバグと誤認しないこと（実行環境起因）。
  同種の注意はサイクルカウンタに限らず、エミュレータが省略しがちな周辺（高精度タイマの
  一部・乱数源・トリム値等）にも通じる。

*〔分類: 観測/エミュレータ ｜ プロセッサ: ARM Cortex-M（例：DWT サイクルカウンタは代表的
エミュレータで常に 0・"非実装"ビットも 0 のまま）／原理は非依存 ｜ 出所: asp3_core〕*
