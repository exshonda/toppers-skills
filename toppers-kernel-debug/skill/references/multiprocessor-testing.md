# マルチプロセッサ（FMP系）派生のテスト・カバレッジ概念（実装非依存）

FMP系（Flexible MultiProcessor）など**SMPカーネル**は、単一プロセッサ（ASP系）に無い
並行・PE間（プロセッサ間）要素を持つ。これが**テスト設計とカバレッジ**に効く。本書は
その差を「なぜ単核と同じテストでは届かない／揺れるのか」という観点で整理する（実装非依存）。

> - 適合性テスト一般の概念（PASS/FAIL/SKIP・FAIL切り分け）は [`conformance-ttsp3.md`](conformance-ttsp3.md)。
> - 実行の観測軸（ディスパッチ・状態遷移・サービスコール出入り）は [`observing-execution.md`](observing-execution.md)。
> - **具体的なターゲット名・割込み番号マクロ・テスト生成器・計測スクリプトはリポジトリ固有**。
>   本書は派生差の概念に徹する。

用語：**PE**＝プロセッシングエレメント（コア）。**IPI**＝プロセッサ間割込み。
**全体ロック（giant lock）**＝カーネル操作をPE間で直列化する大粒度ロック。

## 1. SMPで分岐カバレッジが下がる理由（遅延ディスパッチ枝）

SMPカーネルでは、多くのサービスコールに**「PE間・遅延ディスパッチ」アーム**が追加される。
典型は「全体ロック取得 → 自PEのスケジュール対象タスクが他PEの操作で切り替わっていたら
解放して再ディスパッチ」という分岐。**逐次・単核的に1タスクがAPIを順に呼ぶテストでは、
この分岐は常に成立しない**（他PEが自PEのスケジューリングを変えないため）。

結果として、**SMPの分岐(C1)カバレッジは同型の単核カーネルより低く出る**のが正常で、
コード品質が低いわけではない。対処は2択：

- **実2コアのストレス／インターリーブテストで実証**して到達させる（外部ツール不要で
  実並行を利用する手書きテストが扱いやすい）。
- 到達させられない分は**タイミング依存・到達不能として台帳化**する（残存分岐の分類は
  カバレッジ方法論の領分。ここでは「マルチコア起因」というカテゴリを1つ立てる）。

*〔分類: カバレッジ/SMP ｜ プロセッサ: 非依存（マルチコア一般）｜ 出所: TTSP3(FMP3)〕*

### 1.1 SMPレースを「決定的に」再現する

PE間・遅延ディスパッチ枝の不具合（間欠ハング・想定外アサート）は時間依存で、素朴な
テストでは出たり出なかったりする。**カーネル無改変方針を守ったまま決定的に再現**し、かつ
**観測差を当該修正のみに帰属**させるには、次の二点が効く。

- **最小リバート差分による帰属**：修正済みカーネルから「当該1ファイルの修正だけ」を戻して
  before版を作る。差分が最小になるので、観測した挙動差（ハング/アサート ⇄ 正常）が
  **その修正のみに帰属**することを保証できる。before/after を同梱し別ビルドで再現環境を
  自己完結化すると、第三者が両挙動を確実に再現できる（＝退行の所在をソース1点に絞れる）。
- **リーダレス自己マイグレーション往復**：「発行側タスクが被害側を起こす」構図は同一PE上で
  **優先度スターベーション**を生み、被害側が走れず**偽のハング**になりやすい。代わりに
  **複数タスクが各自を周期的にPE間移送**（自タスクのアフィニティ往復）すると、同一PEの
  優先度衝突なしに**PE間の再ディスパッチ枝を高頻度で駆動**でき、PE間スレッド切替えの競合に
  確実に届く。移送タスクは中位優先度・複数本、制御/集計役は別優先度に分け、busy-loop で
  計測役を飢えさせない（飢餓は偽陽性の温床）。

> ホスト上で擬似実行する移植（タスク＝ホストスレッド）では、この再現法と併せて
> [`host-simulation-port.md`](host-simulation-port.md) のレース窓・per-PE汚染の着眼も使う。

*〔分類: テスト設計/SMPレース再現 ｜ プロセッサ: 非依存（マルチコア一般）｜ 出所: TTSP3(FMP3 POSIXポート)〕*

## 2. 時刻管理＝タイムマスタPEと「時間イベント転送」

SMPカーネルは時刻管理を**タイムマスタPE**に集約し、**時間イベントを処理しないPE**は
自分が登録した時間イベント（タイムアウト・周期・アラーム等）を**マスタへ転送**する設計を取りうる。

- 「時間イベント非処理PE → マスタへ転送」する**転送分岐は、あるPEが時間イベント非処理である
  構成でしか存在しない**。全PEが時間イベント処理に参加する既定構成では、転送分岐は
  **構造的に踏めない**。
- これらに届かせるには、**一部PEを時間イベント非処理にした構成**でテスト／計測する必要がある
  （構成定数のビルド時上書きが要る場合がある。その作法は禁則①の #ifndef ガード／既定不変。
  別skill `toppers-kernel-dev` の `references/inviolable-rules.md` を参照）。

*〔分類: テスト設計/時刻管理 ｜ プロセッサ: 非依存 ｜ 出所: TTSP3(FMP3)〕*

### 2.1 クロスPE時間イベントは「全PE一致の単一時間基準」を要する

別PEに設定したアラーム／タイムアウトの絶対時刻計算と満了判定は、**全PEが同一の時刻基準を
共有する**ことを暗黙の前提にする。**各PEが独立・未同期のハードウェアカウンタで時刻管理する
構成（共有グローバルタイマを持たない）**では、PEごとにカウンタ原点がずれ、クロスPEの満了が
早すぎ／遅すぎになる。

- **通常エミュレータでは顕在化しにくく、実機でのみ落ちる**：エミュレータは全コアがほぼ同時に
  時刻起点へ揃うため基準差≒0だが、実機は**起動順スキュー**（二次コアの解放タイミング差）で
  各カウンタ原点がずれる。「クロスPEアラームのテストがエミュレータで通り実機で落ちる」は
  この基準ずれの典型サイン。
- 対処は、各PEのローカルカウンタをそのまま使わず、**全PEが読む単調なグローバル時刻**を合成する
  （共有タイマがあればそれを基準化し、無ければ基準PEの時刻からのオフセットで各PEを補正する）。
- **対策の核心**：各PEが独立・未同期のタイマしか持たない構成では、**全PEが「タイムマスタ（時刻
  基準を司る1つのPE）」の単一カウンタを共通の時刻基準として読み、そこからグローバルな単調クロック
  を合成**する。これにより全PEが文字通り同一の時刻を見るので、クロスPEの絶対時刻計算・満了判定が
  原点ずれの影響を受けない。折返し直後に時刻が逆行しないよう**直前値でクランプして単調性を保つ**。
  単一のグローバルタイマを全PEが共有するハードでは、この単一時刻基準の条件は**自動的に満たされる**
  ため、合成は per-PE 独立タイマ構成に固有の対策になる。
  *〔分類: 時間/MP・制約/アーキ構造 ｜ プロセッサ: 非依存(per-PE独立タイマ構成で顕在) ｜ 出所: FMP3〕*
- 決定的に検証するなら、シミュレート時刻でPE間バリアを張る手法
  （[`deterministic-time-simulation.md`](deterministic-time-simulation.md)）で基準の同一性を担保する。

*〔分類: テスト設計/時刻管理・制約/アーキ構造 ｜ プロセッサ: 非依存（per-PEタイマで時刻管理するSMP。共有グローバルタイマを持たず ARM Cortex-R5 split 等で顕在）｜ 出所: FMP3(zynqmp_r5)〕*

### 2.2 ソフト拡張した時刻基準は wrap 周期 > 割込み禁止区間 を要する

§2.1 は「PEごとに独立・未同期のカウンタ」を原点ずれの原因にしたが、**同一の共有カウンタ**でも
時刻が凍結する別機構がある。ハードカウンタの幅が時刻表現に足りず、**上位ビットを wrap（折返し）
割込みでソフト更新して拡張**している場合、**割込み禁止／CPUロック区間（スピンロック保持等）が
wrap 周期を超える**と、その間 wrap 割込みが取りこぼされて上位が進まず、ソフト時刻基準が凍結する。

- SMP では**タイムマスタPE**でこれが起きると波及が大きい。マスタが割込み禁止で回っている間、
  他PEが読むグローバル時刻も進まないため、時刻待ち・満了待ちが一斉に止まり**デッドロックに見える**。
- **具体機序（相互待ちデッドロック）**：グローバルな高分解能時刻基準をタイムマスタの周期割込みで
  進める方式では、その周期が短いと、あるPEが割込みを禁止して他PEをスピン待ちさせる区間中に基準の
  更新が止まり（カウンタ折返しと重なって停滞）、**待たれている側の満了（アラーム）が不発**になる。
  すると満了を待つ側も進めず、双方が相手を待つ**相互待ち**に陥る。
- **周期設計の根拠**：この周期は**特定テストの都合で選ばず、実機カウンタのビット幅とクロックを
  根拠に**決める。割込み禁止スピン区間より十分長く取れば、折返しと割込みマスクの重なりが実質ゼロ
  になり、基準の更新停止に起因する不発が起きなくなる。
  *〔分類: 時間/MP・制約/アーキ構造 ｜ プロセッサ: 非依存 ｜ 出所: FMP3〕*
- **通常エミュレータでは顕在化しにくい**：エミュレータ上の割込み禁止区間はモデル時間をほぼ
  消費しないため wrap 周期を跨ぎにくいが、実機では実時間で跨ぐ。
- 対処は、**カウンタ幅に余裕のある実機では全幅を使い、wrap 周期 ≫ 想定最大の割込み禁止区間**に
  すること（狭いカウンタ幅をエミュレータ制約で選んでいる場合は、実機向けに全幅へ切替える）。
  これにより wrap 割込みへの依存区間が実用上発生しなくなる。症状側の切り分けは
  [`debugging-playbook.md`](debugging-playbook.md) §B-14。

*〔分類: 制約/アーキ構造・時刻管理（エミュレータ⇄実機差） ｜ プロセッサ: 非依存（wrap割込みで上位ビットをソフト拡張する時刻基準。狭幅カウンタを実機で全幅化できるアーキで対策可）｜ 出所: FMP3(zynqmp_r5)〕*

## 3. 割込み識別子はPEアフィニティを内包する

SMPでは**割込み番号が対象PE（アフィニティ）とローカルマスクを符号化**しうる。
PE固有割込み（per-processor）と共有割込み（shared）で識別子の意味が異なる。

- 「**別PE宛**の割込み番号」を入力にした異常系テストは、**アフィニティ検証アーム**（自PEと
  対象PEの一致チェックの偽側）を踏む。単に範囲外番号を与えるだけでは、先行句で短絡されて
  この枝に届かないことがある（短絡されない入力＝正しいPE符号化＋不一致アフィニティを選ぶ）。

*〔分類: テスト設計/割込み ｜ プロセッサ: 非依存（割込み番号がPEを符号化する構成）｜ 出所: TTSP3(FMP3)〕*

## 4. 割込みコントローラ方式が異常系テストをゲートする

マルチコアの割込みコントローラ方式は**ローカル / グローバル / 複合**などに分かれ、
**どの割込み異常系テストが適用され、どのエラーコードを期待するか**を左右する。

- ある方式向けに書かれた否定ケースは、**別方式のターゲットでは除外**され実行されない。
- 除外されたテストは**カバレッジにそもそも現れない**（落ちてはいないが踏まれてもいない）。
  「未達」と「非適用」を混同しないよう、**ターゲットの割込み方式ごとに適用テストを監査**する。

*〔分類: テスト設計/割込み ｜ プロセッサ: 非依存（IRC方式差）｜ 出所: TTSP3(FMP3)〕*

## 5. PE間の状態観測には同期点が要る

他PEが起こした状態変化（あるタスクが別PEで実行中になった等）を観測・検証するには、
**IPIの配送や全体ロックの解放が落ち着くのを待つ同期点**を挟む。同期せずに参照すると、
変化前/後のどちらを見るかが非決定的になり、テストがフレークする。

- 実務上は、観測前に**遅延・イベントフラグ・バリア**を1つ入れて、PE間伝搬の完了を保証する。
- 逆に、**PE間状態を参照する分岐**（「対象タスクが他PEで実行中」等）は、同一PE上に置いた
  単核的テストでは踏めない。観測したい分岐に応じて**タスクを別PEへ配置／移送**する。

*〔分類: テスト設計/同期 ｜ プロセッサ: 非依存（マルチコア一般）｜ 出所: TTSP3(FMP3)〕*

### デバッガで一貫スナップショットを取る（全PEを停止）

上は**実行中**の観測の同期点だが、**デバッガで止めて**共有スケジューラ状態（レディキュー・
各PEの実行/予定タスク・優先度ビットマップ）を読むときは、**全PEを停止してから読む**。
1つのPEだけ止めて他PEが走り続けると、全体ロック下で更新途中の共有構造を torn read して
しまい（ビットマップは立っているのにキューが空に見える等）一貫しない。特に**デバッガの
管理外で起動したPE**（PSCI を介さずカーネルが直接起動した副PE等で、デバッガがそのPEを
examine していない場合）は止められないので、観測前にそのPEも examine して停止させる。

*〔分類: 観測/デバッガ・制約/SMP ｜ プロセッサ: 非依存（SMP一般）｜ 出所: fmp3_3.3〕*

## 6. サブ優先度（同一優先度内の並び替え）

SMP系は同一優先度内の順序を**サブ優先度**で制御しうる。

- ある優先度を「サブ優先度有効」に**静的構成**でき、実行可能タスクのサブ優先度変更は
  **同一優先度群内での並び替え**として観測される（レディキュー位置が観測量）。
- **シーリングミューテックスで優先度が昇格**したタスクは、サブ優先度が**最上位にクランプ**される
  （格納値に依らない）。
- これらの分岐は「**サブ優先度を実際に構成し、かつ同一優先度に複数タスクがいる**」場合に初めて
  踏める。既定の自動生成テストが当該優先度にタスクを十分置いていないと未達になりやすい。

*〔分類: テスト設計/スケジューリング ｜ プロセッサ: 非依存（サブ優先度を持つRTOS）｜ 出所: TTSP3(FMP3)〕*

## まとめ：単核との差がテスト設計に効く点

| 差 | 単核テストで起きること | 届かせ方 |
|---|---|---|
| PE間・遅延ディスパッチ枝 | 常に不成立で未達 | 実2コアのインターリーブ実証 or タイミング依存として台帳化 |
| タイムマスタ転送 | 全PE参加構成では構造的に踏めない | 一部PEを時間イベント非処理にした構成 |
| 割込みのPEアフィニティ | 範囲外入力だけでは短絡 | 別PE宛の正しい符号化を入力 |
| 割込みコントローラ方式 | 別方式向け否定ケースは非適用 | 方式ごとに適用テストを監査 |
| PE間状態観測 | 非決定（フレーク）/未達 | 同期点を挟む／タスクをPE間配置 |
| サブ優先度並び替え | 構成不足で未達 | サブ優先度を構成＋同一優先度に複数タスク |

> 具体的な構成定数・割込み番号マクロ・テスト生成器・計測手順は**リポジトリ固有**。
> 本書の概念を、各リポジトリの `AGENTS.md`／`docs/`／専用skillの手順に結びつけて使う。
