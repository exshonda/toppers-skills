# TrustZone-M / Secure境界 dual-OS の診断・観測（実装非依存）

ARMv8-M Security Extension（Secure / Non-secure 二世界）を使う **dual-OS モニタ型**構成
（Secure 側に RTOS、その上で Non-secure OS を起動・監視）を診断・検証するときの、
**実装に依存しない**着眼点。固有のコマンド・番地・ターゲット名・ビルドフラグは書かない
（それらは各リポジトリの `AGENTS.md`／`docs/`／専用skill）。

各項目に出所メタを 〔分類 ｜ プロセッサ ｜ 出所〕 で付す。

---

## 1. セキュアデバッグ・ロック（SWD反復が破綻する正体）
〔制約/アーキ構造＋観測/デバッガ ｜ ARMv8-M Security Extension ｜ 出所: SafeG-M(TrustZone dual-OS)〕

ある M-profile 派生では、**Secure(TrustZone)ファームが一度走ると、外部デバッガからの
ハルト/イグザミネーションが Secure 状態に対して効かなくなる**ことがある（セキュアデバッグ
の許可状態に依存）。症状：接続(DPID 読取り)は成功するのに `halt`/`examine` がタイムアウト
する→書込み(program)に入れない。

- これは故障ではなく**セキュアデバッグの仕様挙動**。SWD 経由の「書込み→実行→再診断」の
  反復は構造的に破綻しやすい。
- 回避：**部品のブートローダ／USB マスストレージ書込み経路（SWD halt 非依存）**で焼く、
  あるいは毎回ブートローダ/リカバリ状態（halt 可能な素の状態）へ物理的に入れてから焼く。
- 「examine は通るが halt できない」を見たら、まずこのロックを疑う（プローブ不良と誤認しない）。

## 2. 常駐デバッガが「意図的例外テスト」を隠す
〔観測/デバッガ ｜ ARM Cortex-M（非依存寄り） ｜ 出所: SafeG-M(NS例外→Secure捕捉テスト)〕

CPU フォルトで停止する**常駐デバッガは、firmware 自身の例外ハンドラが走る前に halt する**。
そのため「わざと CPU 例外を起こして自前ハンドラの捕捉を確認する」種類のテスト
（例：境界越えで意図的に未定義命令→ハンドラで記録）は、**デバッガを接続したままだと
ハンドラに入らず、当該チェックポイントが出力されない**。

- 対策：**デバッガ非接続で自走**させてから観測する（書込み後に切断して reset 自走）。
- 「途中の CP まで出て例外直前で止まる」＝この罠の典型。テストロジックの不具合と誤認しない。

## 3. ワンショット試験出力は capture を reset より先に起動
〔観測/シリアル ｜ 非依存 ｜ 出所: SafeG-M(遷移テスト取得)〕

ループしない（起動時に一度だけ出す）試験出力をシリアルで採るときは、**シリアル capture を
リセット（自走開始）より前に開始**しないと、ブート直後の出力を取り逃す。「0バイト」＝
未出力とは限らず、**捕捉開始が遅れただけ**のことがある。手順を「capture 起動→reset」の
順に固定する。

## 4. 機械可読テストマーカは同期出力にする
〔観測/シリアル＋統合/CI ｜ 非依存 ｜ 出所: SafeG-M([TST]出力)〕

バッファに積んで**低優先のログ出力タスクがドレインする**ログ機構は、テストが
プリエンプト中・別世界(Non-secure)実行中・例外経路にいる間はドレインされず、**該当行が
出ない／取り逃す**。AI/CI が判定する機械可読マーカは、**ログタスク非依存の即時（同期）
低レベル出力**で出すと欠落しない（重複出力にも注意＝1行ちょうど1回に保つ）。

## 5. エミュレータの SG/Secure例外 忠実度の限界
〔観測/エミュレータ＋制約/アーキ構造 ｜ ARMv8-M Security Extension ｜ 出所: SafeG-M(QEMU)〕

エミュレータは SAU・FPU・スタックリミット等を忠実に実装していても、**Secure Gateway(SG)
veneer や Secure 例外まわりの相互作用を実機どおりに再現しないことがある**（例：最初の
secure gate 呼出しで実機には出ない擬似フォルトが決定論的に発生）。

- 症状が**複数のエミュレータ版で同一に再現し、実機では出ない**なら、エミュ固有の限界を疑う
  （カーネル/移植のバグと切り分ける）。
- 運用：**境界を跨ぐ遷移テストの合否は実機を正**とし、エミュレータはビルド健全性・非遷移系
  の確認(CI)に使う。

## 6. FPU 遅延スタッキング残渣が文脈を跨ぐ
〔制約/アーキ構造＋例外 ｜ ARMv8-M（FPU+Security Extension） ｜ 出所: SafeG-M〕

Non-secure→Secure の **gate veneer が FP スクラッチレジスタを使うと、Secure 側の
`CONTROL.FPCA`（FPコンテキスト有り）が立ったまま**になる。その文脈を後で横取りして
コンテキストスイッチすると、ディスパッチャの FP 退避（`vstmdb {s16-s31}` 等）が**想定外の
スタック／スタックリミットでフォルト**しうる。

- 対策の型：**Non-secure を代理する「モニタ/プロキシ的タスク」は FP コンテキストを持たない
  ものとして扱い**、ディスパッチャの FP 退避/復帰を当該タスクではスキップする。
- 「FP退避命令で HardFault」「特定タスクの横取り切替時だけ落ちる」を見たら、境界 veneer の
  FPCA 残渣を疑う。RTOS 側カーネルの FP 文脈管理の最重要回帰項目になりやすい。
