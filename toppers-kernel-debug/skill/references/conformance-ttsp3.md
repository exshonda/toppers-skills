# TTSP3 適合性テスト（概念・実装非依存）

**TTSP3**（TOPPERS Test Suite Package 3）は TOPPERS が提供する**適合性テストスイート**。
カーネルの API／SIL（システムインタフェース層）が仕様どおり振る舞うかを網羅検証する。
ASP3／FMP3／HRP3／HRMP3 等が対象。新ターゲットの移植検証や回帰確認に使う。

## 何を検証するか

| 区分 | 内容 |
|---|---|
| API テスト（functional） | 各サービスコール・状態遷移・待ち/起床・優先度・時間管理等を多数の小テストで検証。各テストは「全チェックポイント通過」で PASS |
| 静的API（staticAPI） | `.cfg` の静的APIが正しく解釈され、不正構成で適切なエラーコードを返すか（cfgエラー系を含む） |
| SIL | システムインタフェース層（メモリ/IOアクセス・割込み管理）の適合 |

## 結果の読み方（PASS / FAIL / SKIP）

- **PASS**：当該ターゲットで期待どおり（チェックポイントを全通過）。
- **DONE（判定点を持たない完走）**：チェックポイントによる合否判定を持たないテストが
  最後まで走り切った状態。計測系（実行時間分布・転送スループット等）や意図的スキップ系が
  該当する。**FAIL ではない**＝合否判定対象でないだけなので、集計で FAIL に混ぜない。
  SKIP（実行を見送り）とも別物で、DONE は**実行はした**。
- **SKIP**：そのテストが要求する**HW機能が当該ターゲットに無い/未対応**で実行を見送った
  もの。例：HWタイマの停止/加速制御、テスト用の割込み生成、CPU例外の人為発生。
  **多くは想定内**で、ターゲット能力の差を表す。
- **FAIL**：仕様不適合の可能性。ただし切り分けが必要（下記）。
- **無出力＝停止（HANG）**：結果行が一切出ない＝プログラムが止まった可能性。FAIL（明示エラーを
  出して完走）とは区別する。停止は症状側の切り分けへ（`debugging-playbook.md` §B）。

> **機械可読の結果マーカで集計を安定させる**：上の4状態は人手では読み分けにくく自動集計で
> 取りこぼす。各テストが**最後に1行の機械可読な結果マーカ**（合格／判定なし完走／失敗を区別する
> 固定書式）を出すようにしておくと、無出力（停止）も「マーカが無い」で機械的に検出できる。
> マーカの具体書式・付与箇所はリポジトリ固有。
> <br>*〔分類: 適合性/TTSP3・統合/CI ｜ プロセッサ: 非依存 ｜ 出所: FMP3〕*

## FAIL の切り分け

FAIL が出たら「真の不具合」か「ターゲット依存の許容差」かを見極める：

1. **複数ターゲットで同じFAILが一致するか**。同一テストが全ターゲットで同じ差分を
   出すなら、ターゲット非依存＝**カーネル/コンフィギュレータ層の差**（例：cfg エラー
   コードの差）であることが多い。1ターゲットだけで出るなら、そのターゲット依存部
   （割込み/例外/タイマ/コンテキスト）を疑う。
2. **staticAPI のエラーコード差**：cfgが不正構成に返すエラーコードが、TTSP3 の期待
   （参照ターゲット基準）と割込み/例外モデルの違いでズレることがある。仕様適合の
   範囲内の差なら**既知のターゲット差として記録**し、functional 適合性とは分けて扱う。
3. **functional の FAIL は重く見る**。API/状態遷移の不適合はカーネル本体の問題の可能性。

## 移植検証での使い方

- 新ターゲットを起こしたら、まずカーネル基本項目の動作確認 → 次に TTSP3 を回して
  網羅的に確認する、という順が定石。
- HW依存テストは SKIP で構わない（ターゲットがその機能を持たないだけ）。
- **gate（合否基準）の例**：functional の FAIL=0（HW依存は SKIP 許容）。staticAPI の
  FAIL は既知差分の一覧と一致すること。

## 実行方法について

TTSP3 自体は Make ベースのテストソース。**TTSP3 を改変せず読み取り専用で使い**、各
プロジェクトのビルドシステム（QEMU/実機）で回す外部ドライバ方式が扱いやすい。
**具体的な起動コマンド・ドライバはリポジトリ固有**（各リポジトリの `docs/`／専用skill）。
