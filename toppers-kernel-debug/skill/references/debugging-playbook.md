# デバッグ・プレイブック（症状→原因→確認）

ITRON系RTOS（ASP/ASP3 等）で頻出する症状の切り分け。実装非依存。

## A. ハードフォルト／CPU例外で停止

**疑う**：スタックオーバーフロー／不正な関数ポインタ・ベクタ／メモリ保護違反
（MPU/PMP/スタックリミット PSPLIM）／非整列アクセス／未定義命令。

**確認**：
1. **CPU例外ハンドラの発生**を観測（どの例外番号か）。
2. フォルトステータスレジスタ（アーキ依存：ARM-M の CFSR/HFSR、AArch64 の ESR、
   RISC-V の mcause/mtval 等）と、フォルト時の PC/スタックを確認。
3. **スタックサイズ**を疑う：タスク/非タスク用スタック（`DEF_ICS`）が不足していないか。
   再帰・大きなローカル配列・深い呼出しが無いか。
4. 例外発生**直前の最後のトレースイベント**から、どの処理で落ちたか特定。
5. ベクタテーブル/リンカ配置の整列要件（例：ARMv8-M の VTOR。整列量はベクタ
   エントリ数×4 以上の2のべき）違反が無いか。**バナー等は出るが最初のタスク切替で
   即死**、が整列違反の典型。移植直後ならここが定番。
   <br>*〔分類: ベクタ/整列 ｜ プロセッサ: ARM Cortex-M（VTOR） ｜ 出所: asp3_mcuxsdk〕*
   - 補足：**ベクタベースの整列違反でも、共通の割込み入口を持つ実装では外部割込み
     （例外番号の高い側）は動作を継続しがち**で、壊れるのはシステム例外（番号の低い側：
     SVC/PendSV/SysTick 等）に偏る。**「ログ・バナーは出るのにタスク切替や tick で死ぬ」**は
     この偏りの典型サイン（割込み経路は動くので発見が遅れる）。
     <br>*〔分類: ベクタ/整列 ｜ プロセッサ: ARM Cortex-M（共通割込み入口） ｜ 出所: asp3_stm32cube〕*
6. **ARMv8-M（Cortex-M/TrustZone）で最初のディスパッチ直後に即フォルト（CFSR の
   INVPC＝不正な PC ロード）** → コンテキスト復帰に使う例外リターン値（EXC_RETURN）の
   Secure/状態ビットが、実際のブート状態（Secure で起動したか）と不整合。Secure 起動なら
   TrustZone を有効化した構成に合わせる。**移植時はまずブート経路（Secure か）を確認**する。
   <br>*〔分類: ブート/例外 ｜ プロセッサ: ARMv8-M（TrustZone） ｜ 出所: asp3_mcuxsdk〕*
7. **割込み禁止区間（ARM-M の PRIMASK 等）で起きた例外は HardFault に昇格**する。
   例外を意図的に起こすテスト（不正命令・不正アクセス系）が、その区間では本来の
   UsageFault でなく HardFault として観測されるのは想定内（真の不具合と切り分ける）。
   <br>*〔分類: 例外/フォルト ｜ プロセッサ: ARM Cortex-M（PRIMASK） ｜ 出所: asp3_mcuxsdk〕*
8. **CPU例外ハンドラ（`cpuexc`）は登録した例外にしか入らない**。未登録の致命フォルト
   （上位フォルトへ昇格した／そもそも捕捉対象にしていない種別）は素通りするため、
   **「ハンドラに来ない＝例外は起きていない」ではない**。発生の有無は、ハンドラ到達でなく
   フォルトステータスや停止状態（デバッガ）で判断する。
   <br>*〔分類: 例外/フォルト ｜ プロセッサ: ARM Cortex-M（cpuexc 登録単位） ｜ 出所: asp3_stm32cube〕*
9. **周期割込み（タイマ等）でスタック破壊・暴走する／割込みハンドラが終わらない** →
   セキュア／高速割込み線（FIQ 相当）でその割込みを配送し、**ハンドラが割込み
   コントローラの ack／EOI より前に同じ割込みを再許可**していないか。ack 前に再許可すると
   同一要因が即座に再入し、ネストでスタックを食い潰して暴走する。RTOS の CPU ロックが
   通常割込み線（IRQ 相当）のマスクで実現されている構成では、**セキュア割込みを通常
   割込み線で配送**する方がロックモデルと整合し、この再入を避けられる。
   <br>*〔分類: 例外・ベクタ/割込み・制約 ｜ プロセッサ: 2本の割込み線＋セキュア分離を持つアーキ（例：ARM GIC＋TrustZone） ｜ 出所: stm32mp2_baremetal〕*
10. **Security Extension を持つコアを Secure 状態だけで動かす移植で、例外復帰の整合性
    検査に失敗してフォルト**（復帰時に状態の食い違いが検出される） → 例外復帰値に
    **Secure 状態を示す値**を選んでいるか。dual-OS（Secure＋Non-secure 併存）でなく
    Secure 単独構成でも、例外復帰値の状態指定は必須で、ここを誤ると復帰時の整合性検査に
    引っ掛かる。項6（dual-OS での状態ビット不整合）の Secure 単独版で、起こる症状は同じ。
    <br>*〔分類: 制約/アーキ構造・例外 ｜ プロセッサ: ARMv8-M ｜ 出所: FMP3(Cortex-M)〕*
11. **セキュアFW（前段ブートローダ）から起動した直後、カーネルが特定の制御レジスタへ
    触れた瞬間に同期例外**（不正命令／トラップ種別の例外） → セキュアFWが **FP/SIMD・
    トレース等へのアーキ的トラップを有効のまま**カーネルへ引き渡していないか。トラップが
    残っていると、当該機能の制御レジスタにアクセスした命令でトラップ例外が起きる。**セキュア
    →カーネルの遷移処理でトラップを解除してから**それらレジスタへ触れる。項6（状態ビット
    不整合）と同様、移植時はブート経路（前段FWが何を設定/トラップ済みか）をまず確認する。
    <br>*〔分類: ブート/例外・制約/アーキ構造 ｜ プロセッサ: 例外レベル＋セキュア分離を持つアーキ（例：AArch64 EL3→EL1） ｜ 出所: fmp3_3.3〕*

## B. 何も起きない／ハング

**疑う**：スケジューラ未起動／全タスクが待ち／割込み（tick）未許可／初期化で停止。

**確認**：
1. **最初のディスパッチ（実行開始タスク）が出るか**。出ない＝起動前で止まっている
   → 初期化ルーチン（`ATT_INI`）・スタートアップ・クロック設定を疑う。
2. **時間が進んでいるか**（tick）。`dly_tsk`/周期ハンドラが全く動かない＝タイマ割込み
   未設定・割込み優先度マスクが全遮断のまま。
3. **全タスクが WAITING** で誰も起こさない＝設計上のデッドロック。待ち条件の供給元
   （`wup_tsk`/`set_flg`/`sig_sem`/`snd_dtq`）が呼ばれる経路を確認。
4. CPUロック（`loc_cpu`）/ディスパッチ禁止（`dis_dsp`）の**解除漏れ**。
5. **バナーやごく初期のログまでは出てそこで止まる（データ／bss 初期化の前後で死ぬ）**
   → キャッシュ／MMU を有効化する**前**の区間は、メモリがデバイス型として扱われ、
   **キャッシュ前提のブロックゼロ化命令や非整列アクセスを使うライブラリ `memset`/`memcpy`
   が例外になる**ことがある。bss/data 初期化を整列ループの自前実装にする。さらに
   **コンパイラが整列ループを `memset`/`memcpy` 呼出しにイディオム認識で戻す**ため、
   `volatile` 等で抑止しないと自前実装が無効化される（逆アセンブルで `bl memset` が
   残っていないか確認）。
   <br>*〔分類: ブート/初期化 ｜ プロセッサ: キャッシュ持ち一般（ブロックゼロ化命令は ARM の DC ZVA が典型） ｜ 出所: stm32mp2_baremetal〕*
6. **時間分割（タイムウィンドウ）で周期オーバランが連発する／時間分割が進まない**
   → 高分解能タイマと時間分割タイマを **1 本の比較タイマに多重化**している場合の典型。
   **割込みがレベルセンスのアーキでは、割込みコントローラの ack 前にコンパレータを
   将来／過去へ動かすと満了イベントが消失・再発**し、時間分割の切替が永久に進まない
   ライブロックになる（ラッチ式のイベントフラグを持つタイマでは起きない）。満了未処理を
   フラグで保持して割込み線をアサートし続け、ack 後にハンドラで満了側をデマルチプレクス
   して処理を完結させる。
   <br>*〔分類: 時間分割/タイマ ｜ プロセッサ: レベルセンス割込み＋比較タイマ一般（例：ARM Generic Timer） ｜ 出所: stm32mp2_baremetal〕*
7. **ごく初期で無出力のままハングする（バナーも出ない／出てすぐ止まる）が、CPU 例外も
   出ていない** → 初期化対象のペリフェラルの**機能クロックやそのディバイダが既定で停止
   （クロックゲート／HALT）のまま**だと、ドライバが最初のレジスタへ書込んだ瞬間に**バスが
   ストール**してハングすることがある（フォルトに昇格せず無音で止まる）。項1（クロック設定）
   の具体形で、「クロックが間違っている」ではなく「クロックが供給されていない（ゲート／
   分周が停止）」を疑う。ドライバ初期化の**前に対象ペリフェラルへのクロック供給を有効化**する。
   <br>*〔分類: 観測/デバッガ・制約/アーキ構造 ｜ プロセッサ: 非依存（クロックゲーティングを持つ SoC 一般。近年の ARM-M SoC で顕在化しやすい） ｜ 出所: asp3_mcuxsdk〕*
8. **完全リセットを伴わず新カーネルを再ロード起動する運用で、時間イベント系だけが
   決定論的にハング**（周期ハンドラ／アラーム／タイムアウト／発火待ちが進まないが、
   同期系・即時系のサービスコールは通る） → 前回実行が割込みコントローラに残した
   **割込みアクティブ状態が消えず、以降その割込みが配送されない**のを疑う。アクティブな
   割込みより同優先以下が遮断されるため、当該割込み（典型はタイマ）由来の時間イベントだけが
   止まり、割込みに依存しない処理は通る——という**症状の偏り**が手掛かり。起動時の割込み
   コントローラ初期化で**アクティブ状態も明示的にクリア**する（クリーンリセット時は無害）。
   <br>*〔分類: ブート/再ロード・症状→原因 ｜ プロセッサ: 非依存 ｜ 出所: FMP3(Cortex-M)〕*

## C. タスクがディスパッチされない／走り続ける

**疑う**：優先度設定／`act_tsk` 漏れ／優先度逆転／ノンプリエンプティブ区間。

**確認**：
1. 対象タスクの状態が DORMANT＝未起動（`act_tsk`/`TA_ACT` 漏れ）。
2. 高優先タスクが READY なのに走らない＝低優先タスクが `dis_dsp`/`loc_cpu`/高い
   割込み優先度マスク（`chg_ipm`）で占有、または**優先度逆転**（C項目）。
3. 同優先度で順番が想定と違う＝レディキューは FCFS。`rot_rdq` の有無を確認。

## D. 優先度逆転

**疑う**：共有資源をセマフォで保護している（優先度継承が効かない）。

**確認**：
1. 共有資源の排他に**ミューテックス**を使い、属性が優先度継承（`TA_INHERIT`）または
   優先度上限（`TA_CEILING`）になっているか。**セマフォは優先度継承しない**ため、
   低優先タスクが資源を握ったまま中優先タスクに割り込まれると高優先が待たされる。
2. 優先度上限ミューテックスなら上限優先度の設定が妥当か。

## E. サービスコールが想定外の `E_*` を返す

| エラー | 典型原因 | 確認 |
|---|---|---|
| `E_CTX` | 非タスクコンテキスト（割込み/ハンドラ）で待ち系API（`wai_*`/`slp_tsk`/`dly_tsk`）を呼んだ、またはタスクから `i`系を呼んだ | 呼出コンテキストとAPIの `〔T〕/〔I〕/〔TI〕` を突合 |
| `E_PAR` | 引数が範囲外（優先度・サイズ・時間値・属性ビット） | 値域を確認 |
| `E_ID` | 存在しない/範囲外のオブジェクトID | `.cfg` の生成と ID 定義を確認 |
| `E_OBJ` | オブジェクトが操作可能な状態にない（二重起床待ち等） | 対象の現在状態 |
| `E_RLWAI` | `rel_wai`/`ter_tsk` 等で待ちが強制解除された | 解除を呼んでいる側を特定 |
| `E_QOVR` | キューイング/カウントの上限超過（`act_tsk` 起動要求過多〔ASPは上限1〕、`wup_tsk` 多重、セマフォ上限） | 上限値と呼出回数 |
| `E_TMOUT` | ポーリング失敗／タイムアウト | 条件成立タイミング |
| `E_NOEXS`/`E_DLT` | 対象が未生成／待ち中に削除 | 生成順・削除の有無 |

エラーコード値・条件の網羅は `toppers-asp/references/error-codes.md` を正本にする。

## 切り分けの基本手順

1. **再現**：ハードなし（ホストsim/QEMU）で再現できると速い。
2. **観測**：トレース/ログで「最後に正常だったイベント」を特定（`observing-execution.md`）。
3. **二分**：問題区間を `.cfg`・初期化・該当タスク・割込みのどれかに絞る。
4. **最小化**：疑う機能だけの最小構成で切り分ける。
5. **根拠**：直したら必ずビルド＋テスト（機械判定）で確認してから報告する。
