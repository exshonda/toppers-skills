# タイマドライバシミュレータで時間管理を決定的に検証する（実装非依存）

時間管理系（周期ハンドラ／アラーム／タイムアウト／システム時刻）のテストは、実タイマで
回すと実機・通常エミュレータのタイミングに依存して**非決定的**になりやすい。**実タイマで
なくシミュレート時刻を使い、サービスコールで進めた分だけ時刻が進むターゲット**を用意すると、
時間管理系を実機／通常エミュレータ非依存で**決定的に回帰**できる。本書はその考え方（実装非依存）。

> - 実行の観測軸は [`observing-execution.md`](observing-execution.md)、症状切り分けは
>   [`debugging-playbook.md`](debugging-playbook.md)。
> - **具体的なターゲット名・タイマ進行 API・割込み源・ビルド手順はリポジトリ固有**。本書は概念に徹する。

## 仕組み（決定的タイミング）

- 高分解能タイマ（HRT）を実ハードでなく**シミュレートした時刻**として持ち、時刻を進めるのは
  **専用の「時刻を進める呼出」だけ**にする。テストが任意量だけ進めれば、満了・タイムアウトの
  発火タイミングがテスト記述どおりに**再現可能**になる（壁時計・実機ジッタに依存しない）。
- これにより、境界ぴったりの満了・同時満了・順序依存といった、実タイマでは狙って起こしにくい
  シナリオを**確定的に**作れる。

## マルチコアでのクロスPE時刻同期

- マルチコアでは、**「時刻を進める呼出」がプロセッサ間バリアになる**ように作ると、各 PE が
  同じシミュレート時刻基準で進むため、**クロスPE の時刻同期**（タイムマスタへの時間イベント
  転送・他PEで設定した満了の観測など）を決定的に試験できる。

## 移植で効くアーキ依存点

- **シミュレートした HRT のローカル割込みをどう起こすかはアーキ依存**。時刻を進めた結果の
  満了を割込みとして上げる必要があるので、**ソフトウェアでペンディング設定できる割込み源**を
  選ぶ（コアが自分でペンディングを立てられる線を割り当てる）。ここだけはプロセッサ依存で、
  他の仕組み（時刻の進め方・テスト記述）は非依存。
- **その「自分で起こせる割込み源」の形はアーキで分かれる**：割込みコントローラへのソフト
  ペンディング設定／専用のプロセッサ間割込み／コアローカルなソフトウェア割込み、など。さらに
  **一部アーキはタイマ／ソフトウェア割込みのトラップ入口がハードワイヤ**で、通常の割込みハンドラ
  静的登録の枠に乗らない。その場合は**既存のトラップ処理へ加算的にフック**して満了処理を呼ぶ
  （静的登録の代わり）。この加算フックは単核化のとき畳んで消える分岐で、移植時のみ意識すればよい。
  <br>*〔分類: 時間/MP・制約/アーキ構造 ｜ プロセッサ: 非依存（誘発手段がアーキ依存。トラップ入口がハードワイヤなアーキ＝RISC-V で顕在） ｜ 出所: FMP3(simtimer)〕*
