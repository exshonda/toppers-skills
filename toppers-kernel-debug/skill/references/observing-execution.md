# 実行の観測（トレース/ログの着眼点・実装非依存）

TOPPERSカーネルの不具合は、まず**実行イベントを時系列で観測**して切り分ける。
トレースログ機能・`syslog`・ログマクロなど、観測手段と出力形式はカーネル/リポジトリに
依存するが、**何を見るか**は共通。本書はその着眼点。

## 観測すべき軸

### 1. ディスパッチ（誰が実行されているか）

- **最初に実行が始まるタスクが出るか**。出ないなら、スケジューラが起動していない／
  起動対象タスクが `act_tsk`/`TA_ACT` で起動されていない／初期化ルーチンで止まっている。
- 切り替えの列が**優先度順**になっているか。高優先タスクが READY なのに低優先が走り
  続けるなら、ディスパッチ禁止（`dis_dsp`）・CPUロック・割込み優先度マスクを疑う。

### 2. タスク状態遷移

μITRON のタスク状態：
- **RUNNING**（実行）／**READY＝RUNNABLE**（実行可能）／**WAITING**（待ち）／
  **SUSPENDED**（強制待ち）／**WAITING-SUSPENDED**（二重待ち）／**DORMANT**（休止）。

遷移を追い、期待した起床・待ち・優先度変更が起きているか確認する：
- ずっと WAITING → 待ち条件が成立していない（誰も `wup_tsk`/`set_flg`/`sig_sem` しない、
  データキューに誰も送らない 等）。
- ずっと DORMANT → そのタスクが `act_tsk` されていない。
- SUSPENDED が解けない → `sus_tsk` と `rsm_tsk` の対応漏れ。

### 3. サービスコールの出入りと復帰値

- 入口はあるのに**出口が無い** → そのAPI内で待ちに入った（正常）か、ハング/フォルトで
  戻れていない。直後に例外イベントが無いか確認する。
- **エラー復帰**（`E_*`＝負値）→ どのAPIが何のコードを返したかを特定し、原因を
  `toppers-asp/references/error-codes.md` と突き合わせる。

### 4. ハンドラの出入り

- 割込みハンドラ/ISR・周期ハンドラ・アラームハンドラ・CPU例外ハンドラの出入り。
- **CPU例外ハンドラが発生している** → ハードフォルト系。`debugging-playbook.md` へ。
- 周期/アラームが**鳴っていない** → `sta_cyc`/`sta_alm` 忘れ、または時間管理（tick）が
  進んでいない（タイマ割込み未設定）。

## 読み解きのコツ

- **時刻（tick/μs）付き**で見て、停止・ハングの「直前の最後のイベント」を特定する。
- 期待イベント列をあらかじめ用意し、実際の列と**機械的に照合**できると回帰検出が速い
  （照合スクリプトの有無・形式はリポジトリ依存）。
- システムログ（`syslog` の COMMENT/ASSERT）はトレースと別経路でコンソールへ出ることが
  多い。アサート失敗メッセージはフォルトの一次情報になる。

## 実機で動作中のカーネルを観測する（OS Awareness 等）

実行中の状態をデバッガで覗くときの実装非依存の注意：

- **動作中の状態はリセットなしの attach 接続で観測する**。通常の接続/書込みはリセットや
  停止を伴い、観測したい状態を壊す（再現性のない初期化途中を見てしまう）。
- **デバッガ（フロントエンド）側の「実行中/停止」状態と、デバッグ機構（オンチップ
  デバッグ／プローブ）側の halt 状態を一致させてから読む**。スクリプトでバックグラウンド
  継続させたあとフロントエンドの割込みで止める手順より、**プローブ（モニタ）側で halt →
  レジスタキャッシュを破棄 → 読む**方が確実。両者がずれると「実行中だから読めない」で
  OS-awareness のメモリ/レジスタ読出しが拒否される。
  <br>*〔分類: 観測/デバッガ ｜ プロセッサ: 非依存 ｜ 出所: asp3_core〕*
- **デバッガのメモリマップに RAM のバスエイリアス（同一メモリの別アドレス窓）が
  欠けていると、その窓に置かれたカーネルオブジェクトの読出しが「アクセス不可」で
  拒否される**ことがある。デバッガ側のメモリアクセス制限を緩める設定が要る場合がある
  （具体的なオプション名・デバイス定義はリポジトリ/ツール固有）。

  *〔分類: 観測/デバッガ ｜ プロセッサ: 非依存（デバッガ・デバイス定義依存） ｜ 出所: asp3_mcuxsdk〕*

- **メモリ保護が有効なとき、デバッガの CPU 経由メモリ読出しは「現在の保護ドメインの
  権限」に従う**。非特権タスクで停止中はカーネル領域や他ドメイン領域が「アクセス不可」で
  読めないことがある（バスエイリアス欠落〔上〕とは原因が別＝権限による拒否）。停止位置の
  特権レベル／実行ドメインを意識して読む（特権文脈で止めれば全域読める）。逆に、
  **非特権で止めるとカーネル領域が読めない＝ドメイン別の保護が実際に効いている証拠**でも
  あり、観測の制約を保護動作の検証に転用できる。

  *〔分類: 観測/デバッガ ｜ プロセッサ: 保護機構付き一般（保護下のメモリ読出しの権限従属） ｜ 出所: stm32mp2_baremetal〕*

## カーネルに OS-awareness 機構が無いとき自作する（実装非依存）

デバッガ側の OS-awareness プラグインが無くても、カーネルの**静的生成テーブルと制御
ブロックをデバッガから読む**ことで、タスク・同期/通信オブジェクト・割込みの一覧を
自作できる（gdb のスクリプト機構などで）。実装非依存の要点：

- **静的情報（初期化ブロック）は `const` ＝ ライブ接続なしに実行ファイルから読める**。
  起動番地・属性・初期優先度・容量・初期値などはROM化される定数なので、ターゲットへ
  繋がずとも実行ファイル単体で一覧できる。一方、**動的情報（制御ブロック：現在状態・
  カウント・キュー・現在優先度）は RAM 上**にあり、実行中ターゲットまたはコアダンプへの
  接続が要る。
  <br>*〔分類: 観測/デバッガ ｜ プロセッサ: 非依存 ｜ 出所: fmp3_3.3〕*

- **オブジェクトID ↔ ユーザ定義名はデバッグ情報に無い**。静的APIで付けた名前は
  コンフィギュレータが生成する定数（マクロ）で、通常のデバッグ情報には載らない。名前を
  出したいときは**生成されたコンフィグ（ID 定義）を併せて読む**。無ければ ID 番号で代替。
  <br>*〔分類: 観測/デバッガ・静的生成 ｜ プロセッサ: 非依存 ｜ 出所: fmp3_3.3〕*

- **キュー/状態の読み解き**。レディキュー・待ちキューは制御ブロックのキューノードで繋がる。
  **キューノードが制御ブロックの先頭にある実装では、ノードのアドレスをそのまま制御ブロック
  として解釈**できる（オフセット 0 の前提を型情報で確認する）。状態は内部表現フィールドを
  デコードし（実行可能/待ち/強制待ち/休止と待ち要因）、制御ブロックのアドレスはテーブル内の
  位置から ID へ逆引きする。
  <br>*〔分類: 観測/デバッガ ｜ プロセッサ: 非依存 ｜ 出所: fmp3_3.3〕*

- **レディキューのビットマップは「ビット位置＝優先度」とは限らない**。最高優先度の探索に
  count-leading-zeros 系命令を使う実装では、ビットマップが**最上位ビット詰め（優先度に対し
  ビット反転）**になっていることがある。デバッガで「どの優先度にタスクが居るか」を読むときは、
  その実装のビット↔優先度対応に合わせる。
  <br>*〔分類: 観測/デバッガ・制約/アーキ構造 ｜ プロセッサ: 非依存（CLZ命令を持つアーキで顕著） ｜ 出所: fmp3_3.3〕*

- **割込みの許可/禁止・ペンディングは割込みコントローラのレジスタを読む**。カーネルの管理
  情報ではなくICの現在状態を見たいときは、許可セット/ペンディングセットレジスタの該当
  ビットを読む。**レジスタ番地はボード/SoC依存なので、観測コードをボード/SoC/コアで層化**して
  可搬部（レジスタ配置の知識）と固有部（ベース番地）を分けると派生間で使い回せる。MMU有効でも
  周辺が device 領域にマップされていれば読める（landing pad の項と同じ前提）。
  <br>*〔分類: 観測/デバッガ ｜ プロセッサ: 非依存（番地は固有→層化） ｜ 出所: fmp3_3.3〕*

- **スタック使用量は保存スタックポインタから概算**できる（スタック上端 − 保存SP）。非実行
  タスクは切替時の保存SPで正確、実行中タスクは最後の切替時の値（概算）。**高水位マーク
  （最大使用量）はスタックをパターンで埋めてある前提**で、埋めない実装では取れないので現
  使用量の概算に留める。
  <br>*〔分類: 観測/デバッガ ｜ プロセッサ: 非依存 ｜ 出所: fmp3_3.3〕*

- **gdb スクリプトで OS-awareness を組むなら Python 対応の gdb が要る**。クロス開発環境
  同梱の gdb は Python 非対応のことがあり、その場合は別途 Python 対応の gdb を用いる
  （どの gdb を使うかはホスト側ツールの事情＝リポジトリ固有）。
  <br>*〔分類: 観測/デバッガ（ツール依存） ｜ プロセッサ: 非依存（ホスト側ツール） ｜ 出所: fmp3_3.3〕*

> SMP で全PEの一貫スナップショットを取る注意は [`multiprocessor-testing.md`](multiprocessor-testing.md)
> 「デバッガで一貫スナップショットを取る」を参照。具体のコマンド・スクリプト・レジスタ番地は
> リポジトリ固有（各リポジトリの専用skill/`docs`）。

## 外部FWが初期化した土台へロードして反復する（landing pad）

外部のブートFW（DRAM 初期化等を行う）から起動するベアメタル構成では、不揮発媒体を
毎回書き換えずに反復するため、**DRAM 初期化済み・キャッシュ無効で停止する最小ブート
イメージ（landing pad）**を一度だけ書いておき、以降はデバッガでカーネル／アプリを
その上へロードする。観測の段取りとして実装非依存の要点：

- **landing pad は MMU／キャッシュを有効化しない状態で停止させる**。キャッシュ無効の
  まま転送すれば、デバッガ（CPU 外のバスマスタ）による書込みとコアのフェッチが
  コヒーレントで、ロード直後の実行が壊れない（キャッシュ有効状態へロードすると、
  古い行が残って未更新コードを実行しうる）。
- ロード後は **PC をロードしたイメージのエントリへ明示設定**してから実行する
  （landing pad の停止位置のまま resume しない）。
- **外部FWにデバッガ待ち合わせ機構が無い**と、接続/リセット時にデバッガがFW固有の
  同期領域へ書こうとして「メモリ書込み失敗」を出すことがあるが、これは無害なことが
  多い（デバッガ設定側でリセットイベントに握り潰しを仕込める）。トレース/カーネルの
  不具合と誤認しない。
  <br>*〔分類: 観測/デバッガ・ブート ｜ プロセッサ: 非依存（外部FWでDRAM初期化するベアメタル一般） ｜ 出所: stm32mp2_baremetal〕*

- **本番ブートローダが設定した周辺の配線（ピンマルチプレクサ）・クロックに依存する
  ベアメタル構成では、デバッガのリセット（特に初期化シーケンスを伴うリセット）がその設定を破壊し、
  コンソール等の周辺が無音になる**。RTOS 側が自前の周辺初期化（HAL の system 起動相当）を
  持たず前段ブートローダの設定に依存する場合は、**リセットを伴わずに 停止→ロード→PC設定→
  実行** とし、前段の周辺設定を保持する。landing pad〔上〕が「DRAM 初期化済みの土台」を
  要点にするのに対し、本件は**周辺の IO 配線・クロック**が要点（機構が別）。なお
  オンチップ SRAM／スクラッチパッドがイメージ全体に足りるなら、**そこへデバッグロード
  すれば DRAM 初期化段自体を不要にできる**（landing pad すら要らない）。
  <br>*〔分類: ブート・観測/デバッガ ｜ プロセッサ: 非依存（前段FWが周辺IO/クロックを設定するベアメタル一般。RISC-V SoC で顕在化） ｜ 出所: asp3_core〕*

- **マルチコア／マルチハート SoC で RTOS を1コアに固定起動する構成では、起動コードが
  他コアをパークする**。このとき**デバッガが列挙するコア索引は、アーキテクチャ上のコア／
  ハート識別子と 1:1 で対応するとは限らない**（デバッグトランスポート経由の識別子レジスタ
  読みも当てにならないことがある）。目的のコアを確実に起動するには、**全コアにエントリ
  番地を設定してから実行**するのが堅実（パーク中のコアは即戻る）。
  <br>*〔分類: 観測/デバッガ・制約/アーキ構造 ｜ プロセッサ: RISC-V マルチハート等（デバッガのコア索引≠アーキのコア識別子） ｜ 出所: asp3_core〕*

## 実機でフォルトの瞬間を捕まえる（デバッガ）

ハードフォルト/ハングの一次情報は、**発生の瞬間に停止させて**集める。停止後に状態を
追いかけても、二次的に走った別フォルト・ループに惑わされる。

- **デバッガのベクタキャッチ（各フォルト種別で自動停止）を仕掛けて発生瞬間で止める**。
  止めたら、フォルトステータス・フォルトアドレス・スタック上の例外フレーム（積まれた
  復帰 PC 等）・被フォルト文脈の例外番号を読む（症状対応は `debugging-playbook.md`）。
- **「被フォルト文脈の例外番号」と「停止 PC が属する関数」が矛盾したら、ベクタテーブルの
  配置（整列）不正を疑う**決定的サイン。例：ある例外番号で入ったはずなのに PC が別の
  ハンドラ内＝ベクタフェッチがずれている。
  <br>*〔分類: 観測/デバッガ ｜ プロセッサ: ARM Cortex-M（ベクタキャッチ／例外番号） ｜ 出所: asp3_stm32cube〕*
- **バックトレースを鵜呑みにしない**。リンカの同一コード畳み込み（ICF：終端の `while(1)`
  群などが同一アドレスへ畳まれる）で、`addr2line`/デバッガが**無関係な関数名を表示**する
  ことがある。シンボル名でなく、**PC の生アドレス・フォルトステータス・スタック上の
  例外フレーム**を一次情報にする。
  <br>*〔分類: 観測/デバッガ ｜ プロセッサ: 非依存（リンカ ICF 最適化） ｜ 出所: asp3_stm32cube〕*
- デバッガから**低レベル出力関数を直接呼び、出力経路だけを単体検証**できる（停止中でも
  ポーリング実装なら出力される）。「出ない＝カーネル不具合」と早合点しない。
- **アイドルでコアクロックを止める省電力命令を使う構成では、停止中のコアにオンチップ
  デバッグが応答せず、コアを止め直せない**（halt 要求がクロック停止中のコアに届かない）。
  計測・デバッグビルドはアイドルを**クロックを止めないビジーアイドル**へ差し替えると確実に
  止められる。差し替えは**計測ビルド専用にガード**し、製品ビルドは通常どおり省電力にする。
  <br>*〔分類: 観測/デバッガ・制約/アーキ構造 ｜ プロセッサ: 非依存 ｜ 出所: FMP3(Cortex-M)〕*

## トレースが出ないとき

- トレース機能が**無効ビルド**になっている → 有効化して再ビルド。
- 実機で**文字出力先が未設定** → 低レベル文字出力（ターゲット依存の put 関数）の実装を確認。
  低レベル出力（カーネルの put 関数・起動前/CPUロック中/例外文脈で使うログ）は、
  **libc の stdio や `_write`/`putchar` 経由でなく、周辺（UART 等）レジスタの直接ポーリングで
  実装する**のが要点。stdio 経由だと未実装の下位フック（`_write` 等）に握り潰され「一切出ない」に
  なりやすく、そもそもカーネル起動前・割込み禁止中・例外中では stdio が使えない。直接ポーリングなら
  全文脈で出るため、診断の最終手段として確実。
  <br>*〔分類: 観測/シリアル ｜ プロセッサ: 非依存（低レベル出力の実装方針） ｜ 出所: asp3_stm32cube〕*
- 実機シリアル（UART/USB-CDC 等）の出力が**欠落・文字化け・無出力**に見えるときは、
  同じシリアルデバイスを**複数プロセスが同時に開いていないか**をまず疑う（リーダは1つに
  限る。複数だとバイトを奪い合う）。トレース/カーネルの不具合と誤認しやすい観測側の罠。
  <br>*〔分類: 観測/シリアル ｜ プロセッサ: 非依存（ホスト側） ｜ 出所: asp3_mcuxsdk〕*
- 実機で**末尾の出力だけが欠ける**（テストのサマリ行・終了メッセージが出ない）ときは、
  カーネル/アプリの**終了処理が送信FIFOのドレインを待たずにシリアルを停止/無効化**して
  いないか疑う。送信はFIFO経由で非同期なので、停止前に「送信完了（busy 解除）」を待た
  ないと最後の数十バイトが捨てられる。**エミュレータは送信が瞬時なので顕在化せず、実機で
  のみ出る**（実機・エミュレータ差で誤認しやすい）。
  <br>*〔分類: 観測/シリアル ｜ プロセッサ: 非依存 ｜ 出所: asp3_core〕*
- エミュレータで**特定ボード／起動方式が丸ごと無出力**のときは、移植バグと即断する前に
  **エミュレータのバージョンがそのマシン/ブート方式に対応しているか**を確認する。直接
  ブート（前段ファームウェア無しで実イメージを起動する方式等）は新しめのエミュレータで
  ないと無対応のことがあり、古い版では沈黙する。新しい版・ピン留めした実行環境
  （コンテナ等）・実機のいずれかで切り分ける。
  <br>*〔分類: 観測/エミュレータ(QEMU) ｜ プロセッサ: 非依存 ｜ 出所: asp3_core〕*
- 実機で**全コアがパーク状態・識別子読みが固定値・無出力が続く**ときは、**JTAG／デバッグ
  アダプタが短時間の多数の 停止/ロード/リセット 反復で応答不能（wedge）になっている**ことを
  疑う。**プローブの電源再投入（再接続）で前段ブートからやり直すと復旧**する。カーネル／
  トレースの不具合と誤認しない観測側の罠。
  <br>*〔分類: 観測/デバッガ（ホスト/プローブ依存） ｜ プロセッサ: 非依存（ホスト側JTAGプローブ一般） ｜ 出所: asp3_core〕*
- **UART の配線・分周が不確実な実機立ち上げでは、デバッガ経由のセミホスティング出力を
  結果取得チャネルにする**と UART 非依存に合否を採れる（シリアルが鳴る前から結果が読める）。
  マルチコアでは**出力する全コアでセミホスティングを有効化**する——コア毎にデバッガ側で
  ハンドラ処理が要るため、複数コアが出力するバリア等で一部コアだけ未有効だと、そのコアの
  出力で停滞して全体が進まない。
  <br>*〔分類: 観測/デバッガ ｜ プロセッサ: 非依存 ｜ 出所: FMP3(Cortex-M)〕*
