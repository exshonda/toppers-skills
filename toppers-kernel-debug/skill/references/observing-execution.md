# 実行の観測（トレース/ログの着眼点・実装非依存）

TOPPERSカーネルの不具合は、まず**実行イベントを時系列で観測**して切り分ける。
トレースログ機能・`syslog`・ログマクロなど、観測手段と出力形式はカーネル/リポジトリに
依存するが、**何を見るか**は共通。本書はその着眼点。

## 観測すべき軸

### 1. ディスパッチ（誰が実行されているか）

- **最初に実行が始まるタスクが出るか**。出ないなら、スケジューラが起動していない／
  起動対象タスクが `act_tsk`/`TA_ACT` で起動されていない／初期化ルーチンで止まっている。
- 切り替えの列が**優先度順**になっているか。高優先タスクが READY なのに低優先が走り
  続けるなら、ディスパッチ禁止（`dis_dsp`）・CPUロック・割込み優先度マスクを疑う。

### 2. タスク状態遷移

μITRON のタスク状態：
- **RUNNING**（実行）／**READY＝RUNNABLE**（実行可能）／**WAITING**（待ち）／
  **SUSPENDED**（強制待ち）／**WAITING-SUSPENDED**（二重待ち）／**DORMANT**（休止）。

遷移を追い、期待した起床・待ち・優先度変更が起きているか確認する：
- ずっと WAITING → 待ち条件が成立していない（誰も `wup_tsk`/`set_flg`/`sig_sem` しない、
  データキューに誰も送らない 等）。
- ずっと DORMANT → そのタスクが `act_tsk` されていない。
- SUSPENDED が解けない → `sus_tsk` と `rsm_tsk` の対応漏れ。

### 3. サービスコールの出入りと復帰値

- 入口はあるのに**出口が無い** → そのAPI内で待ちに入った（正常）か、ハング/フォルトで
  戻れていない。直後に例外イベントが無いか確認する。
- **エラー復帰**（`E_*`＝負値）→ どのAPIが何のコードを返したかを特定し、原因を
  `toppers-asp/references/error-codes.md` と突き合わせる。

### 4. ハンドラの出入り

- 割込みハンドラ/ISR・周期ハンドラ・アラームハンドラ・CPU例外ハンドラの出入り。
- **CPU例外ハンドラが発生している** → ハードフォルト系。`debugging-playbook.md` へ。
- 周期/アラームが**鳴っていない** → `sta_cyc`/`sta_alm` 忘れ、または時間管理（tick）が
  進んでいない（タイマ割込み未設定）。

## 読み解きのコツ

- **時刻（tick/μs）付き**で見て、停止・ハングの「直前の最後のイベント」を特定する。
- 期待イベント列をあらかじめ用意し、実際の列と**機械的に照合**できると回帰検出が速い
  （照合スクリプトの有無・形式はリポジトリ依存）。
- システムログ（`syslog` の COMMENT/ASSERT）はトレースと別経路でコンソールへ出ることが
  多い。アサート失敗メッセージはフォルトの一次情報になる。

## 実機で動作中のカーネルを観測する（OS Awareness 等）

実行中の状態をデバッガで覗くときの実装非依存の注意：

- **動作中の状態はリセットなしの attach 接続で観測する**。通常の接続/書込みはリセットや
  停止を伴い、観測したい状態を壊す（再現性のない初期化途中を見てしまう）。
- **デバッガのメモリマップに RAM のバスエイリアス（同一メモリの別アドレス窓）が
  欠けていると、その窓に置かれたカーネルオブジェクトの読出しが「アクセス不可」で
  拒否される**ことがある。デバッガ側のメモリアクセス制限を緩める設定が要る場合がある
  （具体的なオプション名・デバイス定義はリポジトリ/ツール固有）。

  *〔分類: 観測/デバッガ ｜ プロセッサ: 非依存（デバッガ・デバイス定義依存） ｜ 出所: asp3_mcuxsdk〕*

## 実機でフォルトの瞬間を捕まえる（デバッガ）

ハードフォルト/ハングの一次情報は、**発生の瞬間に停止させて**集める。停止後に状態を
追いかけても、二次的に走った別フォルト・ループに惑わされる。

- **デバッガのベクタキャッチ（各フォルト種別で自動停止）を仕掛けて発生瞬間で止める**。
  止めたら、フォルトステータス・フォルトアドレス・スタック上の例外フレーム（積まれた
  復帰 PC 等）・被フォルト文脈の例外番号を読む（症状対応は `debugging-playbook.md`）。
- **「被フォルト文脈の例外番号」と「停止 PC が属する関数」が矛盾したら、ベクタテーブルの
  配置（整列）不正を疑う**決定的サイン。例：ある例外番号で入ったはずなのに PC が別の
  ハンドラ内＝ベクタフェッチがずれている。
  <br>*〔分類: 観測/デバッガ ｜ プロセッサ: ARM Cortex-M（ベクタキャッチ／例外番号） ｜ 出所: asp3_stm32cube〕*
- **バックトレースを鵜呑みにしない**。リンカの同一コード畳み込み（ICF：終端の `while(1)`
  群などが同一アドレスへ畳まれる）で、`addr2line`/デバッガが**無関係な関数名を表示**する
  ことがある。シンボル名でなく、**PC の生アドレス・フォルトステータス・スタック上の
  例外フレーム**を一次情報にする。
  <br>*〔分類: 観測/デバッガ ｜ プロセッサ: 非依存（リンカ ICF 最適化） ｜ 出所: asp3_stm32cube〕*
- デバッガから**低レベル出力関数を直接呼び、出力経路だけを単体検証**できる（停止中でも
  ポーリング実装なら出力される）。「出ない＝カーネル不具合」と早合点しない。

## トレースが出ないとき

- トレース機能が**無効ビルド**になっている → 有効化して再ビルド。
- 実機で**文字出力先が未設定** → 低レベル文字出力（ターゲット依存の put 関数）の実装を確認。
  低レベル出力（カーネルの put 関数・起動前/CPUロック中/例外文脈で使うログ）は、
  **libc の stdio や `_write`/`putchar` 経由でなく、周辺（UART 等）レジスタの直接ポーリングで
  実装する**のが要点。stdio 経由だと未実装の下位フック（`_write` 等）に握り潰され「一切出ない」に
  なりやすく、そもそもカーネル起動前・割込み禁止中・例外中では stdio が使えない。直接ポーリングなら
  全文脈で出るため、診断の最終手段として確実。
  <br>*〔分類: 観測/シリアル ｜ プロセッサ: 非依存（低レベル出力の実装方針） ｜ 出所: asp3_stm32cube〕*
- 実機シリアル（UART/USB-CDC 等）の出力が**欠落・文字化け・無出力**に見えるときは、
  同じシリアルデバイスを**複数プロセスが同時に開いていないか**をまず疑う（リーダは1つに
  限る。複数だとバイトを奪い合う）。トレース/カーネルの不具合と誤認しやすい観測側の罠。
  <br>*〔分類: 観測/シリアル ｜ プロセッサ: 非依存（ホスト側） ｜ 出所: asp3_mcuxsdk〕*
