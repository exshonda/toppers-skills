# 実行の観測（トレース/ログの着眼点・実装非依存）

TOPPERSカーネルの不具合は、まず**実行イベントを時系列で観測**して切り分ける。
トレースログ機能・`syslog`・ログマクロなど、観測手段と出力形式はカーネル/リポジトリに
依存するが、**何を見るか**は共通。本書はその着眼点。

## 観測すべき軸

### 1. ディスパッチ（誰が実行されているか）

- **最初に実行が始まるタスクが出るか**。出ないなら、スケジューラが起動していない／
  起動対象タスクが `act_tsk`/`TA_ACT` で起動されていない／初期化ルーチンで止まっている。
- 切り替えの列が**優先度順**になっているか。高優先タスクが READY なのに低優先が走り
  続けるなら、ディスパッチ禁止（`dis_dsp`）・CPUロック・割込み優先度マスクを疑う。

### 2. タスク状態遷移

μITRON のタスク状態：
- **RUNNING**（実行）／**READY＝RUNNABLE**（実行可能）／**WAITING**（待ち）／
  **SUSPENDED**（強制待ち）／**WAITING-SUSPENDED**（二重待ち）／**DORMANT**（休止）。

遷移を追い、期待した起床・待ち・優先度変更が起きているか確認する：
- ずっと WAITING → 待ち条件が成立していない（誰も `wup_tsk`/`set_flg`/`sig_sem` しない、
  データキューに誰も送らない 等）。
- ずっと DORMANT → そのタスクが `act_tsk` されていない。
- SUSPENDED が解けない → `sus_tsk` と `rsm_tsk` の対応漏れ。

### 3. サービスコールの出入りと復帰値

- 入口はあるのに**出口が無い** → そのAPI内で待ちに入った（正常）か、ハング/フォルトで
  戻れていない。直後に例外イベントが無いか確認する。
- **エラー復帰**（`E_*`＝負値）→ どのAPIが何のコードを返したかを特定し、原因を
  `toppers-asp/references/error-codes.md` と突き合わせる。

### 4. ハンドラの出入り

- 割込みハンドラ/ISR・周期ハンドラ・アラームハンドラ・CPU例外ハンドラの出入り。
- **CPU例外ハンドラが発生している** → ハードフォルト系。`debugging-playbook.md` へ。
- 周期/アラームが**鳴っていない** → `sta_cyc`/`sta_alm` 忘れ、または時間管理（tick）が
  進んでいない（タイマ割込み未設定）。

## 読み解きのコツ

- **時刻（tick/μs）付き**で見て、停止・ハングの「直前の最後のイベント」を特定する。
- 期待イベント列をあらかじめ用意し、実際の列と**機械的に照合**できると回帰検出が速い
  （照合スクリプトの有無・形式はリポジトリ依存）。
- システムログ（`syslog` の COMMENT/ASSERT）はトレースと別経路でコンソールへ出ることが
  多い。アサート失敗メッセージはフォルトの一次情報になる。

## トレースが出ないとき

- トレース機能が**無効ビルド**になっている → 有効化して再ビルド。
- 実機で**文字出力先が未設定** → 低レベル文字出力（ターゲット依存の put 関数）の実装を確認。
