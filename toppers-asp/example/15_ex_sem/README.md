# 15_ex_sem — セマフォによる排他制御

## 何のサンプルか

`14_ex_none` で発生していた共有変数の競合を，**セマフォ** で排他制御することで解決する。

学習ポイント：

- **`CRE_SEM`** でセマフォを生成
- **`wai_sem`** で資源獲得（クリティカルセクション開始）
- **`sig_sem`** で資源返却（クリティカルセクション終了）
- 排他制御用セマフォは初期値=1, 最大値=1（バイナリセマフォ）
- セマフォの待ちで自タスクが待ち状態に入り，他タスクに切替可能

## 前サンプルとの差分

| 項目 | 14_ex_none | 15_ex_sem |
|---|---|---|
| 排他制御 | なし | セマフォ `SEM1` |
| `count` 更新前 | 直接アクセス | `wai_sem(SEM1)` |
| `count` 更新後 | 直接アクセス | `sig_sem(SEM1)` |
| 静的API | `CRE_TSK`+`CRE_CYC` | + `CRE_SEM` |

## なぜそう書くか

### タスク

```c
void
task(intptr_t exinf){
    int local_count = 0;

    for (;;) {
        wai_sem(SEM1);                            /* セマフォ獲得 */
        count = busy_wait_inc(count);             /* クリティカルセクション */
        sig_sem(SEM1);                            /* セマフォ返却 */
        local_count = busy_wait_inc(local_count); /* セクション外 */
        syslog(LOG_NOTICE, "Task %d : count = %d, local_count = %d",
               exinf, count, local_count);
    }
}
```

- `wai_sem(SEM1)` 〜 `sig_sem(SEM1)` の間が**クリティカルセクション**。
- このセクションには **同時に1タスクのみ**が入れる（バイナリセマフォ）。
- 他のタスクが `wai_sem` を呼ぶと**待ち状態**に入り，`sig_sem` が呼ばれるまで待つ。

### `asp_prog.cfg`

```
CRE_SEM(SEM1, { TA_NULL, 1, 1 });
```

`CRE_SEM(semid, { sematr, isemcnt, maxsem })`:
- `TA_NULL`: 待ち行列はFIFO順（タスク優先度順なら `TA_TPRI`）
- `isemcnt=1`: 初期資源数1（最初は誰も使っていないので資源1個分が利用可能）
- `maxsem=1`: 最大資源数1（バイナリセマフォ）

### 実行結果（期待値）

```
Task 1 : count = 5, local_count = 4
Task 2 : count = 9, local_count = 4
Task 1 : count = 11, local_count = 5
```

→ `count` の増分は `local_count` の増分の**約2倍**（2タスクが正しくインクリメントしているため）。

## セマフォの動作

| 状態 | `wai_sem` の動作 | `sig_sem` の動作 |
|---|---|---|
| カウンタ > 0 | カウンタ-1 して即座にreturn | カウンタ+1（待ち中タスクなければ） |
| カウンタ = 0 | 自タスクが待ち状態に | 待ち行列の先頭タスクを起こす（カウンタは変化しない） |

このサンプル（バイナリセマフォ）：

1. 初期状態: カウンタ=1
2. Task1 が `wai_sem` → カウンタ=0，Task1がクリティカルセクションに入る
3. Task2 が `wai_sem` → カウンタ=0，Task2は待ち状態
4. Task1 が `sig_sem` → 待ち中の Task2 を起こし，Task2 がクリティカルセクションに入る（カウンタは0のまま）
5. Task2 が `sig_sem` → カウンタ=1（誰も待っていないので）

## カウンティングセマフォの用途

`isemcnt`/`maxsem` を1以上にすると **カウンティングセマフォ** になる。

例：4個のバッファがあり最大4タスクまで同時アクセス可：

```
CRE_SEM(BUF_SEM, { TA_NULL, 4, 4 });
```

排他制御以外の用途：

- **生産者-消費者**: 生産側が `sig_sem`（資源あり通知），消費側が `wai_sem`（資源待ち）
- **イベントカウント**: 割込みハンドラが `isig_sem` でイベント発生数を加算

## ミューテックスとの違い

| | セマフォ | ミューテックス |
|---|---|---|
| 用途 | 排他制御・同期通知両方 | 排他制御専用 |
| 所有者の概念 | なし | あり（ロックしたタスクが解除する） |
| 優先度継承/上限 | なし | あり（`TA_CEILING`属性） |
| ASP動作中の解放 | 任意のタスクが `sig_sem` | ロックしたタスク以外は `unl_mtx` で `E_ILUSE` |

排他制御専用なら **ミューテックス** が推奨（優先度逆転を抑制できる）。`16_sw_flg_cyc` 以降ではミューテックスは出てこないが，本格的な排他制御には `CRE_MTX` + `loc_mtx`/`unl_mtx` を検討すること。

## 他の排他制御方法との比較

| 方法 | 影響範囲 | 割込み応答性 | 推奨用途 |
|---|---|---|---|
| セマフォ | 該当タスク間のみ | 影響なし | タスク間排他（推奨） |
| ミューテックス | 該当タスク間のみ | 影響なし | タスク間排他（優先度逆転対策） |
| ディスパッチ禁止 | 全タスク | 影響なし | タスク間の極短いCS |
| CPUロック | 全タスク + 全割込み | 悪化 | 割込みハンドラとの排他必要時 |
| 割込み優先度マスク | 一部割込み | 一部悪化 | 特定優先度以下の割込みのみ抑止 |

## LLM向け要点

- タスク間排他は **`CRE_SEM` + `wai_sem`/`sig_sem`** または **`CRE_MTX` + `loc_mtx`/`unl_mtx`**。
- 排他制御用バイナリセマフォは `isemcnt=1, maxsem=1`。
- `wai_sem` でタスクが待ち状態に入り，他のタスクが動けるので**割込み応答性は維持**される。
- クリティカルセクション内で **エラー時の `sig_sem` 忘れ**に注意。早期リターン時は明示的に `sig_sem` してから return。
- 割込みハンドラからセマフォを操作するときは **`isig_sem`** を使う。`wai_sem` は割込みハンドラから呼べない（待ちに入るため）。
