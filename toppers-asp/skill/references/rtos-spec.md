# TOPPERS/ASP カーネル RTOS仕様（LLM向けスタイルガイド）

## 0. 本書の位置付け

本書は，TOPPERS新世代カーネル統合仕様書 Release 1.7.1（<https://www.toppers.jp/docs/tech/ngki_spec-171.pdf>，以下「本仕様」）のうち **TOPPERS/ASPカーネル** に関する部分のみを抽出し，アプリケーション開発を支援するLLM向けに構造化したものである。FMP（マルチプロセッサ版）／HRP2（メモリ保護版）／HRMP（保護＋マルチプロセッサ版）／SSP（最小サブセット版）／ASP Safety に固有の機能・規定は対象外とする。

本書では，**オブジェクトの生成は静的API（システムコンフィギュレーションファイル `.cfg` への記述）のみをサポート** する前提で記述する。動的生成機能拡張パッケージ由来のサービスコール（`acre_*`, `del_*`）は範囲外とする。

## 1. ASPカーネルの位置付け

ASP（Advanced Standard Profile）はTOPPERS新世代カーネルの出発点となるリアルタイムカーネルであり，**μITRON 4.0仕様をベースとしつつ拡張・改良を加えた実装仕様**である。完全互換ではなく，主要な点で仕様が異なるため，μITRON 4.0準拠の文献やコードをそのまま流用するとビルドエラーや実行時エラーになることがある。差分の概要は次節「1.1 μITRON 4.0からの差分」を参照。

主な特徴：

- 高い信頼性・安全性・リアルタイム性を要求される組込みシステムを主対象（プログラムサイズで数十KB〜1MB程度）。
- カーネル内で**動的なメモリ管理を一切行わない**。**すべてのカーネルオブジェクトは静的APIで生成する**（`.cfg` ファイルに記述）。動的生成API（`acre_*` / `del_*`）は標準パッケージでは**未サポート**。
- 保護機能・マルチプロセッサ・動的生成のいずれにも対応しない（ベース構成）。
- ただし固定長メモリプール機能はサポートし，アプリケーションが動的にメモリ確保することは可能。

ASPカーネルは保護機能対応カーネルではないため，本書ではメモリオブジェクト管理・拡張サービスコール管理・アクセス許可ベクタ・保護ドメイン・カーネルドメイン／ユーザドメインに関する記述は省略する。同様にマルチプロセッサ対応カーネルではないため，スピンロック・クラス・プロセッサIDに関する記述も省略する。

### 1.1 μITRON 4.0からの差分

ASPカーネルは μITRON 4.0仕様をベースとしているが，以下の点で仕様が異なる。LLMが μITRON 4.0準拠のコードを書こうとすると ASP では動かないため，必ず ASP仕様に従うこと。

#### A. データ型の見直し（C99準拠化）

| 項目 | μITRON 4.0 | ASP |
|---|---|---|
| サイズ固定整数型 | `B`, `H`, `W`, `UB`, `UH`, `UW` 等 | `int8_t`, `int16_t`, `int32_t`, `uint8_t` 等（C99） |
| 処理単位のエントリ番地 | `FP` 型（パラメータが反映されない） | `TASK`, `TEXRTN`, `CYCHDR`, `ALMHDR`, `INTHDR`, `EXCHDR`, `INIRTN`, `TERRTN`, `ISR` 等 |
| スタック領域 | （特定型なし） | `STK_T` 型 |
| 固定長メモリプール領域 | （特定型なし） | `MPF_T` 型 |
| 割込みハンドラ番号型 | 実装依存 | `INHNO`（`uint_t`） |
| 割込み番号型 | 実装依存 | `INTNO`（`uint_t`） |
| CPU例外ハンドラ番号型 | 実装依存 | `EXCNO`（`uint_t`） |

μITRON 4.0 のITRON標準データ型（`B`/`H`/`W`/`UB`/`UH`/`UW` 等）が必要な場合は `itron.h` を明示的にインクルードする（標準ヘッダ `kernel.h` には含まれない）。

#### B. 値が0の属性・定数の廃止／値変更

ASP では「値=0 の属性」は**廃止され「指定しない＝デフォルト動作」に統一**された。μITRON 互換のために `itron.h` には定義が残るが，ASPアプリケーションでは使わない：

| 廃止された属性 | μITRONでの意味 | ASPでの正しい記述 |
|---|---|---|
| `TA_TFIFO`(=0x00) | 待ち行列FIFO順 | 何も指定しない（属性が他にないなら `TA_NULL`） |
| `TA_MFIFO`(=0x00) | メールボックスのメッセージFIFO順 | 何も指定しない |
| `TA_HLNG`(=0x00) | 高級言語インタフェース | 何も指定しない |
| `TA_WSGL`(=0x00) | 単一タスクのみ待ち | 何も指定しない（複数待ちは `TA_WMUL` を明示） |

また値が0だった動作モード／状態定数も値変更されている：

- `TWF_ANDW`（イベントフラグAND待ち）: μITRONでは0 → ASPでは0x02
- `TTEX_ENA` 等のオブジェクト状態: 値が変更されている

#### C. 廃止されたサービスコール／マクロ

μITRON 4.0 にあったが ASP仕様からは廃止されたもの。`itron.h` でマクロ定義（互換用）または完全廃止：

| 廃止 | 経緯 | 代替 |
|---|---|---|
| `frsm_tsk` | 強制待ち要求ネスト機能を廃止 | `rsm_tsk` を使う（`itron.h` で `frsm_tsk` を `rsm_tsk` にマクロ定義） |
| `TMAX_SUSCNT` | 強制待ち要求ネスト数の最大値 | 1に固定（`itron.h` に定義移動） |
| `set_tim` | システム時刻の設定 | 完全廃止（代替なし） |
| `isig_tim` | システム時刻通知 | カーネル内部処理化（サービスコールとしては廃止） |
| `t_services.h` / `s_services.h` | μITRON 4.0 のヘッダ構成 | 廃止。ASPでは `kernel.h` または `sil.h`（共通部は `t_stddef.h`） |

#### D. 動的生成APIは標準パッケージでサポート外

| 動的生成系 API | μITRON 4.0 | ASP標準パッケージ |
|---|---|---|
| `acre_tsk`, `acre_sem`, `acre_flg`, `acre_dtq`, `acre_mbx`, `acre_mtx`, `acre_mpf`, `acre_cyc`, `acre_alm`, `acre_isr` | 標準で利用可 | **未サポート**（動的生成機能拡張パッケージ専用） |
| `del_tsk`, `del_sem`, `del_flg`, `del_dtq`, `del_mbx`, `del_mtx`, `del_mpf`, `del_cyc`, `del_alm`, `del_isr` | 標準で利用可 | **未サポート**（動的生成機能拡張パッケージ専用） |
| `AID_*` 静的API | 標準で利用可 | **未サポート**（動的生成機能拡張パッケージ専用） |

**ASP標準パッケージでは，すべてのカーネルオブジェクトを静的API（`.cfg` 上の `CRE_*` / `DEF_*` / `ATT_*`）で生成する**。動的にオブジェクトを増減するアプリは ASP標準パッケージでは作れない。

#### E. 仕様変更されたAPI

| API | μITRON 4.0 | ASP |
|---|---|---|
| `ext_tsk` | 非タスクコンテキストから呼ぶ動作は実装依存 | 非タスクコンテキストから呼ぶと **`E_CTX`**（戻り値型は `ER`） |
| `sta_cyc` | 呼出後最初の起動は「起動周期」で指定した相対時間後 | 呼出後最初の起動は「**起動位相**」で指定した相対時間後 |
| タスク例外処理ルーチンの実行開始 | 割込み優先度マスク状態に依存しない | 割込み優先度マスク全解除でない場合は実行開始されない |
| CPU例外ハンドラから呼べるAPI | μITRON 4.0仕様 | TOPPERS標準割込み処理モデルに準拠（`xsns_dpn`/`xsns_xpn` で判定） |

#### F. 管理領域の分離（生成パケットの変更）

ASP では一部のオブジェクトでカーネル管理領域を分離する仕様が追加された：

| 静的API | 追加／変更パラメータ |
|---|---|
| `CRE_MPF` | `mpfmb`（固定長メモリプール管理領域の先頭番地）を追加 |
| `CRE_DTQ` | 最後のパラメータ名を `dtqmb`（データキュー管理領域の先頭番地）に変更 |
| `CRE_MBX` | 互換性維持のため**変更なし**（管理領域分離なし） |

`mpfmb`/`dtqmb` に `NULL` を指定するとカーネルが自動確保するので，多くの場合 `NULL` で良い。

#### G. ASP独自の拡張機能（μITRON 4.0にない／JSPカーネル由来）

ASPでは μITRON 4.0 にない以下のサービスコール・静的APIが追加されている：

| API | 機能 |
|---|---|
| `get_utm` | 性能評価用システム時刻参照（μ秒単位） |
| `ext_ker` | カーネルの終了 |
| `sns_ker` | カーネル非動作状態の参照 |
| `get_inf` | 自タスクの拡張情報の参照 |
| `xsns_dpn` / `xsns_xpn` | CPU例外ハンドラ用の状態参照 |
| `chg_ipm` / `get_ipm` | 割込み優先度マスクの変更／参照 |
| `dis_int` / `ena_int` | 割込みの禁止／許可（ターゲット非依存に標準化） |
| `ini_sem` / `ini_flg` / `ini_dtq` / `ini_mbx` / `ini_mpf` | 同期・通信オブジェクトの再初期化 |
| 優先度データキュー一式 | `CRE_PDQ`, `snd_pdq`, `psnd_pdq`, `ipsnd_pdq`, `tsnd_pdq`, `rcv_pdq`, `prcv_pdq`, `trcv_pdq`, `ini_pdq`, `ref_pdq` |
| `ATT_TER` | 終了処理ルーチンの追加（静的API） |
| `DEF_ICS` | 非タスクコンテキスト用スタック領域の定義（静的API） |
| `CFG_INT` | 割込み要求ラインの属性設定（静的API） |
| `ATT_ISR` | 割込みサービスルーチンの追加（静的API） |
| `ref_*` 系 | デバッグ用にすべての主要オブジェクトの状態参照を実装 |

#### H. インクルード構成の変更

| アプリ用途 | μITRON 4.0 | ASP |
|---|---|---|
| カーネル機能のみ | `t_services.h` | `kernel.h`（自動的に `t_stddef.h` も取込） |
| SIL機能のみ | `s_services.h` | `sil.h` |
| カーネルもSILも使わない | （該当なし） | `t_stddef.h` |
| μITRON 4.0互換型・定数（`B`/`H`/`W`, `TA_TFIFO`, `frsm_tsk` 等）が必要 | （標準で取込） | `itron.h` を明示的にインクルード |

#### I. 静的APIのみでオブジェクト生成（再強調）

ASP標準パッケージでは：

- すべてのタスク・セマフォ・イベントフラグ・データキュー・優先度データキュー・メールボックス・ミューテックス・固定長メモリプール・周期ハンドラ・アラームハンドラ・割込みサービスルーチン・割込みハンドラ・CPU例外ハンドラ・初期化ルーチン・終了処理ルーチンは，**静的API（`.cfg` ファイル中の `CRE_*` / `DEF_*` / `ATT_*` / `CFG_INT`）で生成**する。
- 実行時にオブジェクトを生成・削除する API は**存在しない**。
- 同様に，**`AID_*`（ID自動割付）も標準では未サポート**。識別名は `kernel_cfg.h` 経由で参照する。

これは μITRON 4.0 と異なる点で，「動的にタスクを増やしたい」「ランタイムにセマフォを作りたい」といった μITRON 流の発想は ASP では使えない。代わりに「すべてのオブジェクトを `.cfg` で予め宣言し，必要に応じて `act_tsk`/`ter_tsk`，`ini_*`，`sus_tsk`/`rsm_tsk` 等で状態を制御する」というスタイルになる。

#### 既存μITRONコード移植時の典型的なエラー

| 症状 | 原因 | 対処 |
|---|---|---|
| `TA_TFIFO is undeclared` | μITRON 4.0属性を直接使用 | 削除して `TA_NULL` に，または `itron.h` インクルード |
| `acre_tsk is undeclared` | 動的生成API使用 | `.cfg` の `CRE_TSK` に置換 |
| `del_sem is undeclared` | 動的削除API使用 | 静的生成のため不要。必要なら `ini_sem` を検討 |
| `frsm_tsk is undeclared` | 廃止API使用 | `rsm_tsk` に置換（または `itron.h` インクルード） |
| `set_tim is undeclared` | 廃止API使用 | 代替なし。設計を見直す |
| `B`, `H`, `W` の型エラー | μITRON 標準型を使用 | `int8_t`/`int16_t`/`int32_t` に置換，または `itron.h` インクルード |
| `t_services.h not found` | 旧ヘッダを使用 | `kernel.h` に置換 |
| `void task(VP_INT exinf)` のシグネチャ警告 | μITRON型を使用 | `void task(intptr_t exinf)` に置換 |

## 2. APIの構成要素とコンベンション

### 2.1 APIの種類

- **サービスコール（service call）**: 上位のソフトウェアからカーネルを呼び出すC関数インタフェース。
- **コールバック**: カーネルから上位のソフトウェアを呼び出すインタフェース。タスク本体・周期ハンドラ等の処理単位が該当する。
- **静的API（static API）**: システムコンフィギュレーションファイル `.cfg` に記述するインタフェース。オブジェクトの生成情報や初期状態を定義する。
- **構成マクロ**: `TMAX_*`, `TMIN_*`, `TNUM_*` 等，カーネルや機能の上限値・サイズ等を表すC言語マクロ。

### 2.2 命名規則（ネーミングコンベンション）

- サービスコール名は `xxx_yyy` 形式。`xxx` は操作（例: `act`, `wai`, `sig`），`yyy` は対象（例: `tsk`, `sem`, `flg`）。
- 非タスクコンテキスト専用は先頭に `i` を付加（例: `iact_tsk`, `isig_sem`）。
- ポーリング版は先頭に `p`，タイムアウト版は先頭に `t`（例: `pol_sem`, `twai_sem`）。
- 静的API名はサービスコール名の英小文字を英大文字に置換したもの（例: `CRE_TSK` ↔ `acre_tsk`）。ただしASPでは静的APIは `CRE_*`／`DEF_*`／`ATT_*`／`AID_*` 等の形を取り，動的生成系の `acre_*`／`del_*` はサポート外。
- データ型は `T_C*`（生成パケット），`T_R*`（参照パケット），`T_D*`（定義パケット）等のT_接頭辞付き型，または `*_T` 接尾辞付き型。
- ポインタ変数は `p_*`（ポインタ），`pk_*`（パケットへのポインタ），`pp_*`（ポインタへのポインタ）。

### 2.3 返値とエラーコード

サービスコールの返値型は `ER`（符号付き整数）。`E_OK`（=0）または正値で正常終了，負値はエラーコードを表す。エラーコードは `E_xxx` 形式。エラー検出条件についてはサービスコール毎に規定される（詳細は `error-codes.md`）。

ID取得型のサービスコールは `ER_ID`，符号無し整数を返すものは `ER_UINT` を返値型に持つ。`bool_t` を返すサービスコール（`sns_*`, `xsns_dpn`, `xsns_xpn`）は例外的に真偽値を返す。

静的APIには返値はない。コンフィギュレータがエラーを検出した場合はビルド時エラーとなる。

### 2.4 ヘッダファイル

- `kernel.h` — カーネルAPI全般。インクルードすると `t_stddef.h` も自動的にインクルードされる。
- `t_stddef.h` — TOPPERS共通定義（基本データ型・エラーコード・共通マクロ）。
- `kernel_cfg.h` — コンフィギュレータが生成するヘッダ（オブジェクトIDマクロ等）。アプリケーションからは直接インクルードしない。
- `sil.h` — システムインタフェースレイヤ（SIL）API。

`kernel.h` をインクルードする前に `TOPPERS_SVC_CALL` をマクロ定義すると，サービスコール呼出しが関数呼出しコードとして展開される。ASPは保護機能非対応のため，通常はこの設定で問題ない。

## 3. 主要な概念

### 3.1 オブジェクトと処理単位

カーネルが管理対象とするソフトウェア資源を**カーネルオブジェクト**と呼ぶ。ASPでサポートされるカーネルオブジェクトは次の通り：

- タスク
- セマフォ／イベントフラグ／データキュー／優先度データキュー／メールボックス／ミューテックス
- 固定長メモリプール
- 周期ハンドラ／アラームハンドラ
- 割込みサービスルーチン／割込みハンドラ／CPU例外ハンドラ
- 初期化ルーチン／終了処理ルーチン

オブジェクトはID番号で識別する（例外: 割込みハンドラは割込みハンドラ番号，CPU例外ハンドラはCPU例外ハンドラ番号で識別）。ASPでは静的APIにより登録するため，ID番号は `.cfg` に記述したオブジェクト識別名にコンフィギュレータが順に割り付ける。識別名は `kernel_cfg.h` にマクロ定義されるので，アプリケーションは識別名を使ってオブジェクトを参照する。

**オブジェクト属性**は `ATR` 型（符号無し整数）で，`TA_xxx` 定数のビット毎論理和（`|`）で複数指定する。指定する属性がない場合は `TA_NULL`（=0）を指定する。

**処理単位（processing unit）** は，プログラムが対応付けられるオブジェクト（または対応付けられるプログラム）。ASPの処理単位はタスク・タスク例外処理ルーチン・割込みハンドラ・割込みサービスルーチン・周期ハンドラ・アラームハンドラ・CPU例外ハンドラ・初期化ルーチン・終了処理ルーチンである。

**拡張情報（extended information）** は処理単位が呼び出されるときに `intptr_t exinf` パラメータとして渡される値で，処理単位の登録時に指定する。カーネルやシステムサービスの動作には影響しない。

### 3.2 優先度と時間

- **優先度（priority）**: `PRI` 型（符号付き整数）。**値が小さいほど優先度が高い**。タスク優先度は1から連続した正値。割込み優先度は-1から連続した負値（多重割込み制御用）。
- **システム時刻**: `SYSTIM` 型（符号無し整数, 単位ミリ秒）。タイムティック毎に更新。
- **相対時間**: `RELTIM` 型（符号無し整数, 単位ミリ秒）。最大値は構成マクロ `TMAX_RELTIM` に定義。少なくとも `uint16_t` 範囲は格納可能。
- **タイムアウト**: `TMO` 型（符号付き整数, 単位ミリ秒）。
  - 正値: タイムアウトまでの相対時間
  - `TMO_POL`（=0）: ポーリング動作（待たずに失敗で `E_TMOUT`）
  - `TMO_FEVR`（=-1）: 永久待ち（タイムアウトなし）

タイムアウト付きサービスコールがタイムアウトすると `E_TMOUT` が返る。`TMO_POL` 指定時にディスパッチ保留状態から呼び出すと `E_CTX` エラーとなることに注意。

## 4. 処理単位の種類と実行順序

### 4.1 ASPがサポートする処理単位

ASPでは次の処理単位をサポートする：

- タスク（タスク例外処理ルーチンを含む）
- 割込みハンドラ・割込みサービスルーチン
- 周期ハンドラ・アラームハンドラ（タイムイベントハンドラ）
- CPU例外ハンドラ
- 初期化ルーチン・終了処理ルーチン

オーバランハンドラと拡張サービスコールはASPコア仕様ではサポートしない。オーバランハンドラ拡張パッケージを使えば追加可能だが，本書では対象外とする。

### 4.2 優先順位

優先順位は処理単位の実行順序を決める仕様上の概念。複数の処理単位が実行可能なら最も高い優先順位の処理単位が実行される。

- 割込みハンドラの優先順位 > ディスパッチャの優先順位 > タスクの優先順位
- 同じ割込み優先度の割込みハンドラ間は，先に実行開始されたほうが優先順位が高い。
- 同じタスク優先度のタスク間は，先に実行できる状態になったほうが優先順位が高い（FCFS）。
- タスク例外処理ルーチンの優先順位は対象タスクと同じだが，対象タスクより先に実行される。
- CPU例外ハンドラはCPU例外を起こした処理単位と同じ優先順位だが，それよりも先に実行される。
- 初期化ルーチンは `.cfg` 中に記述された順に実行。終了処理ルーチンは記述と逆順に実行。

### 4.3 タスクの状態遷移

タスクは次の状態を取る：

- **実行できる状態**（runnable）: さらに**実行状態**（running）と**実行可能状態**（ready）に分類。
- **休止状態**（dormant）: タスクが実行すべき処理がない状態。実行を再開するための情報は保存されない。
- **広義の待ち状態**（blocked）: 実行を止められている状態。実行再開情報は保存される。
  - **待ち状態**（waiting）: 自ら実行を止めている。
  - **強制待ち状態**（suspended）: 他から強制的に止められている。
  - **二重待ち状態**（waiting-suspended）: 上記2つが重なった状態。

ASPでは未登録状態（non-existent）にはならない（オブジェクトは静的APIで登録され削除されない）。

タスク生成直後は，属性に `TA_ACT` を指定したタスクは実行できる状態，指定しなかったタスクは休止状態になる。`act_tsk` でタスクを起動できる。タスクのメインルーチンからリターンするか `ext_tsk` を呼び出すとタスクは終了し，休止状態に遷移する。

### 4.4 タスクのスケジューリング規則

- 実行できるタスクの中で最も高い優先順位を持つタスクが実行状態となる。
- 同じ優先度のタスクはFCFS（First Come First Served）でスケジューリング。`rot_rdq` で同じ優先度内の優先順位を回転できる。
- 最も高い優先順位のタスクが変化するとディスパッチが起こる（ディスパッチ保留状態でない限り）。

ASPはプリエンプティブ・優先度ベース・FCFSスケジューラと考えてよい。

### 4.5 待ち行列と待ち解除順序

タスクが同期・通信オブジェクトの待ち行列につながれる順序は，オブジェクト属性で指定：

- 何も指定しない: **FIFO順（既定）**
- `TA_TPRI`（=0x01）: タスク優先度順

ASP仕様では値=0の属性 `TA_TFIFO` は廃止された。FIFO順は属性を明示しない（属性が他にないなら `TA_NULL` を指定）。イベントフラグなど待ち解除条件がタスク毎に異なる場合の挙動も属性で指定可能（`TA_WMUL` 等）。詳細は各オブジェクトの仕様を参照。

## 5. システム状態とコンテキスト

### 5.1 動作状態

- **カーネル動作状態**: カーネルの初期化完了後〜終了処理開始前。サービスコールが利用可能。
- **カーネル非動作状態**: 初期化完了前と終了処理開始後。SILのAPIと `sns_ker` のみが呼び出せる。

### 5.2 コンテキスト

- **タスクコンテキスト**: タスク（タスク例外処理ルーチンを含む）が実行されるコンテキスト。
- **非タスクコンテキスト**: 割込みハンドラ・割込みサービスルーチン・タイムイベントハンドラ・CPU例外ハンドラが実行されるコンテキスト。

各サービスコールは呼出可能コンテキストが規定される（記号 `〔T〕`/`〔I〕`/`〔TI〕`）：

- `〔T〕`: タスクコンテキスト専用。非タスクコンテキストから呼ぶと `E_CTX`。
- `〔I〕`: 非タスクコンテキスト専用。タスクコンテキストから呼ぶと `E_CTX`。
- `〔TI〕`: どちらからでも呼べる。

非タスクコンテキスト用に派生したサービスコール名は先頭に `i` が付く（例: `iact_tsk`, `iset_flg`, `irot_rdq`）。両コンテキスト共用の `〔TI〕` API（例: `sns_*`, `loc_cpu`, `unl_cpu`）には `i` 付き版が併存する場合があるので使い分けが必要。

### 5.3 カーネルの振舞いに影響を与える状態

プロセッサは互いに独立な以下の状態を持つ：

- **全割込みロック状態**: NMI以外のすべての割込みをマスク。`SIL_LOC_INT()` / `SIL_UNL_INT()` で制御。SILのAPIと `sns_ker`, `ext_ker` のみ呼び出せる。
- **CPUロック状態**: カーネル管理のすべての割込みをマスク。`loc_cpu`/`iloc_cpu`, `unl_cpu`/`iunl_cpu` で制御。多くのサービスコールが `E_CTX` でエラーになる（例外: `loc_cpu`, `unl_cpu`, `dis_int`, `ena_int`, `sns_*`, `get_utm`, `ext_tsk`, `ext_ker`）。
- **割込み優先度マスク**: `TIPM_ENAALL`（=0）以外の値が設定されている状態を「全解除でない状態」と呼ぶ。`chg_ipm`/`get_ipm` で制御。広義の待ちに入る可能性のあるサービスコールは `E_CTX` でエラー。
- **ディスパッチ禁止状態**: `dis_dsp`/`ena_dsp` で制御。広義の待ちに入る可能性のあるサービスコールは `E_CTX` でエラー。

これらのうち**非タスクコンテキスト実行中・CPUロック状態・割込み優先度マスクが全解除でない状態・ディスパッチ禁止状態**を総称して**ディスパッチ保留状態**と呼ぶ。ディスパッチ保留状態ではディスパッチが起こらない。

### 5.4 処理単位の開始・終了とシステム状態

| 処理単位 | 実行開始条件 | 終了/リターン時処理 |
|---|---|---|
| タスク | CPUロック解除・割込み優先度マスク全解除・ディスパッチ許可 | 終了時にカーネルが解除・全解除・許可に設定 |
| タスク例外処理ルーチン | CPUロック解除・割込み優先度マスク全解除・ディスパッチ任意 | リターン時にCPUロック解除，割込みマスク全解除，ディスパッチフラグを元に戻す |
| カーネル管理の割込みハンドラ／ISR／タイムイベントハンドラ | CPUロック解除・割込み優先度マスクは自優先度より低い・ディスパッチフラグ任意 | 実行開始時に割込み優先度マスクを自優先度に設定，リターン時に元に戻す |
| CPU例外ハンドラ | 任意 | リターン時にCPUロックフラグと割込み優先度マスクを元に戻す |

要するに，アプリケーションコードがCPUロックや割込み優先度マスクを変更した場合，原則として処理単位から戻る前にアプリケーションが元に戻す責任がある。タスクが終了するときはカーネルが強制的にデフォルト状態へ戻す。

## 6. 割込み処理モデル

### 6.1 割込み処理の流れ

周辺デバイス→割込み要求ライン→割込みコントローラ（IRC）→プロセッサ。プロセッサが割込み要求を受け付けると，**カーネル管理の割込みであれば**カーネル内の割込み入口処理を経由してカーネル内の割込みハンドラが実行され，その中でアプリケーションが登録した**割込みサービスルーチン（ISR）** が呼び出される。

ISRは特定のデバイスのみに依存して書くべきで，プロセッサや割込みコントローラに依存しないのが原則。1つの割込み要求ラインに複数のISRを登録可能（`ATT_ISR` を複数記述）。

カーネル標準の割込みハンドラで対応できない場合のみ，アプリケーション側で割込みハンドラを定義する（`DEF_INH`）。原則はISRを使うこと。

### 6.2 割込み優先度と多重割込み

- 割込み優先度は `PRI` 型，**負値**で表現（-1が最高）。
- 段階数はターゲット定義。`TMIN_INTPRI`〜`TMAX_INTPRI`（=-1）の範囲で指定。
- 割込み要求受付時，プロセッサは割込み優先度マスクを受け付けた割込み優先度に設定する。これにより同じか低い優先度の割込みは新たに受け付けられず（多重割込み禁止），より高い優先度の割込みのみ受け付けられる。
- 割込み処理からのリターン時にマスクは元の値に戻る。

### 6.3 割込み要求ラインの属性

各割込み要求ラインには次の属性を持たせる（`CFG_INT` で設定）：

- **割込み要求禁止フラグ**: マスク用
- **割込み優先度**: 割込み処理の優先順位を決定
- **トリガモード**: レベル/エッジ等

属性が設定されていない割込み要求ラインは禁止フラグがセットされたまま。`dis_int`/`ena_int` で動的に禁止フラグを操作できる（ターゲット依存）。

### 6.4 割込み番号と割込みハンドラ番号

- **INTNO型**: 割込み番号（割込み要求ラインを識別）。`CFG_INT`, `CRE_ISR`/`ATT_ISR`, `dis_int`, `ena_int` で使う。
- **INHNO型**: 割込みハンドラ番号（アプリ提供の割込みハンドラを登録する対象）。`DEF_INH` で使う。
- 1対1対応するのが基本。両者の値が一致することが多い（ターゲット定義）。

### 6.5 カーネル管理外の割込み

`TMIN_INTPRI` よりも高い割込み優先度の割込みはカーネル管理外。NMIは常にカーネル管理外。カーネル管理外の割込みハンドラからは SILのAPIと `sns_ker`, `ext_ker` のみ呼び出せる（その他のサービスコールは未保証）。

カーネル管理外の割込みハンドラは `DEF_INH` で属性に `TA_NONKERNEL` を指定して登録する。カーネル管理外の割込みにISRは登録できない。

## 7. CPU例外処理モデル

### 7.1 CPU例外ハンドラ

CPU例外は実行中の処理に依存して発生する例外（ゼロ除算・不正アクセス等）。`DEF_EXC` で `EXCNO` を対象にCPU例外ハンドラを登録する。

CPU例外ハンドラからは原則として `xsns_dpn`, `xsns_xpn` のみ呼び出せる：

- `xsns_dpn(p_excinf)` が `false` を返したとき: 非タスクコンテキストから呼べる全サービスコールが利用可能（リカバリ用にタスクを起動・待ち解除する方式に使える）。
- `xsns_xpn(p_excinf)` が `false` を返したとき: 非タスクコンテキストから呼べる全サービスコールが利用可能（CPU例外を起こしたタスクにタスク例外処理を要求する方式に使える）。
- 両方 `true` のときはハードウェア独立な復旧 (a) または「システム再起動」 (d) しか採れない。

### 7.2 カーネル管理外のCPU例外

カーネル非動作状態・カーネル内クリティカルセクション中・全割込みロック状態・CPUロック状態・カーネル管理外の割込みハンドラ実行中のいずれかで発生したCPU例外をカーネル管理外のCPU例外と呼ぶ。これらでは `xsns_dpn`/`xsns_xpn` ともに `true` を返す。

## 8. システムの初期化と終了

### 8.1 初期化手順

1. リセット後，スタートアップモジュールが実行される（モード初期化・スタックポインタ設定・全割込みマスク・bssクリア・dataセクション初期化等）。
2. カーネル自身の初期化処理（内部データ構造の初期化）と静的APIの処理（オブジェクトの登録など）。
3. `ATT_INI` で登録した初期化ルーチンが `.cfg` 記述順に実行される。
4. カーネル動作状態に遷移し，すべての割込みがマスク解除されてタスクの実行が開始される。

### 8.2 終了手順

1. アプリケーションが `ext_ker` を呼び出す。
2. NMIを除くすべての割込みがマスクされ，タスクの実行が停止される。
3. `ATT_TER` で登録した終了処理ルーチンが `.cfg` 記述と**逆順**で実行される。
4. ターゲットシステム依存の終了処理が呼び出される。

### 8.3 初期化ルーチン・終了処理ルーチンの記述形式

```c
void initialize_routine(intptr_t exinf)
{
    /* exinfには登録時の拡張情報が渡される */
    /* タスクコンテキストでも非タスクコンテキストでもないので，
       ほとんどのサービスコールは呼び出せない（カーネル動作状態に遷移する前のため） */
}

void terminate_routine(intptr_t exinf)
{
    /* 終了処理ルーチンも同様 */
}
```

初期化ルーチンはカーネル非動作状態で実行されるので，呼び出せるサービスコールはSILのAPIと `sns_ker` のみ。アプリケーション側の初期化（システムサービスのデバイス初期化等）に用いる。

## 9. オブジェクトの登録（静的APIによる静的生成）

### 9.1 システムコンフィギュレーションファイル `.cfg`

システムの構成を記述するテキストファイル。コンフィギュレータが処理して `kernel_cfg.h` と `kernel_cfg.c` を生成する。記述できるのは：

- カーネルの静的API
- システムサービスの静的API
- C言語プリプロセッサのインクルードディレクティブ（`#include`）と条件ディレクティブ（`#if`, `#ifdef` など）
- コンフィギュレータの `INCLUDE("...");` ディレクティブ

ASPでは保護ドメインの囲み（`DOMAIN`）・クラスの囲み（`CLASS`）は使用しない（保護機能・マルチプロセッサ非対応）。

### 9.2 静的APIの文法

C言語の関数呼出しと同様の文法。構造体パラメータは `{ ... }` で囲む。各APIの末尾に `;` を付ける。

```c
#include "kernel_cfg_target.h"
#include "myapp.h"

CRE_TSK(MAIN_TASK, { TA_ACT, 0, main_task, MAIN_PRIORITY, STACK_SIZE, NULL });
CRE_SEM(SEM_BUTTON, { TA_NULL, 0, 1 });
ATT_INI({ TA_NULL, 0, init_routine });
```

### 9.3 静的APIのパラメータ種別

| 種別 | 説明 | 記述可能な式 |
|---|---|---|
| (a) オブジェクト識別名 | オブジェクトのID番号を指定するパラメータ | 単一の識別名のみ |
| (b) 整数定数式パラメータ | 属性・サイズ・優先度等 | 番地に依存しない整数定数式 |
| (c) 一般定数式パラメータ | 関数ポインタ・拡張情報・先頭番地等 | 任意の定数式 |
| (d) 文字列パラメータ | セクション名・モジュール名等 | C言語の文字列リテラル |

整数定数式パラメータの解釈に必要なヘッダは，`.cfg` から `#include` してインクルードする必要がある。

### 9.4 オブジェクトの登録と解除

ASPは静的APIのみのため，**オブジェクトを生成する静的API（`CRE_TSK`, `CRE_SEM`, `CRE_FLG`, `CRE_DTQ`, `CRE_PDQ`, `CRE_MBX`, `CRE_MTX`, `CRE_MPF`, `CRE_CYC`, `CRE_ALM`, `CRE_ISR`）** によってのみオブジェクトが登録される。同じ識別名のオブジェクトを2回生成すると `E_OBJ` エラー（コンフィギュレータが報告）。

`ATT_ISR`, `ATT_INI`, `ATT_TER` は識別名（ID番号）を持たないオブジェクトを追加する。

ASPでは登録したオブジェクトの登録解除（`del_*` サービスコール）はサポートしない。

## 10. 名前空間とビルド構成

### 10.1 主要な構成マクロ

ASPにおける代表的な構成マクロ：

| マクロ | 意味 |
|---|---|
| `TMAX_ACTCNT` | タスクの起動要求キューイング数の最大値（ASPでは1固定） |
| `TMAX_RELTIM` | 相対時間の最大値 |
| `TMAX_MAXSEM` | セマフォの最大資源数の上限（=`UINT_MAX`） |
| `TMIN_INTPRI`, `TMAX_INTPRI`(=-1) | 割込み優先度の範囲 |
| `TMIN_ISRPRI`(=1), `TMAX_ISRPRI`(=16) | ISR優先度の範囲（ASP既定） |
| `TIPM_ENAALL`(=0) | 割込みをマスクしない値 |
| `TPRI_SELF`(=0) | `chg_pri` で自タスクのベース優先度を表す |
| `TPRI_INI`(=0) | `chg_pri` で起動時優先度を意味する |
| `TSK_SELF`(=0) | サービスコールで自タスクを意味する |
| `TSK_NONE`(=0) | 待ち行列にタスクが存在しないことを表す |

### 10.2 タスクとスタックの記述形式

```c
#include <kernel.h>
#include <t_syslog.h>

void my_task(intptr_t exinf)
{
    while (1) {
        wai_sem(SEM_BUTTON);
        /* 処理 */
    }
    /* 万一ループを抜けてもカーネルが ext_tsk() 同等の終了処理を行う */
}
```

`.cfg` 例：

```c
#include "myapp.h"

CRE_TSK(MAIN_TASK,
        { TA_ACT,                /* 生成と同時に起動 */
          (intptr_t)0x1234,      /* 拡張情報 */
          my_task,               /* タスクのメインルーチン */
          MAIN_PRI,              /* 起動時優先度 */
          STACK_SIZE,            /* スタックサイズ（バイト） */
          NULL });               /* スタック先頭番地（NULLでカーネルが確保） */
```

スタック領域はアプリケーションが確保する場合，`STK_T` 型配列を `STACK_SIZE` に対応するサイズで宣言する。スタックサイズの計算には `COUNT_STK_T(size)` マクロを使用する。

## 11. 仕様適用上の注意（ASPサブセット）

本書の対象（ASPカーネル）から除外したものを再掲する：

- メモリオブジェクト管理機能（`ATT_MEM`, `ATT_PMA`, `att_mem`, `prb_mem` 等）— HRP2系
- 拡張サービスコール管理機能（`DEF_SVC`, `cal_svc`, `def_svc`）— HRP2系
- スピンロック（`CRE_SPN`, `loc_spn` 等）— FMP系
- マルチプロセッサ系API（`mact_tsk`, `mig_tsk`, `msta_cyc`, `msta_alm`, `mrot_rdq`, `get_pid` 等）— FMP系
- アクセス許可ベクタ系API（`SAC_*`, `sac_*`）— 保護機能対応カーネル
- ID番号自動割付け（`AID_*`）— 動的生成機能拡張パッケージ
- 動的生成系（`acre_*`, `del_*` のサービスコール版）— 動的生成機能拡張パッケージ
- オーバランハンドラ（`DEF_OVR`, `sta_ovr`, `stp_ovr`）— ASP拡張パッケージ
- 制約タスク（`TA_RSTR` 属性, `DEF_EPR`）— SSP固有・ASP拡張パッケージ
- メッセージバッファ（`CRE_MBF`）— HRP2系
- TA_PHS属性の周期ハンドラ — ASPでは未サポート

これらをアプリケーションコードに含めてはならない。LLMがコードを生成する場合，上記APIをユーザに提案する前に「ASPの標準パッケージのみ」かを確認すべきである。

## 12. 推奨スタイル（LLM出力規約）

LLMがTOPPERS/ASP向けコードを生成するとき，以下を遵守する：

1. **ヘッダは `kernel.h` を最初にインクルード**。`t_stddef.h` は自動的にインクルードされる。
2. **すべてのオブジェクトは `.cfg` の静的APIで生成**。動的生成APIをアプリ本体に含めない。
3. **オブジェクトIDはコンフィギュレータが生成するマクロ名を使う**（例: `CRE_TSK(MAIN_TASK, ...)` で宣言された `MAIN_TASK` をサービスコールに渡す）。
4. **コンテキストを意識して `i` 接頭辞付きAPIを選ぶ**。割込みハンドラ／ISR／周期ハンドラ／アラームハンドラからは `iact_tsk`, `iset_flg`, `isig_sem` 等を使用する。
5. **タスクのメインルーチンは無限ループまたは `ext_tsk` 終了**。リターンしてもカーネルが終了処理するが，明示が望ましい。
6. **エラーコードは符号付き整数で受ける（`ER`/`ER_ID`/`ER_UINT`）**。`E_OK` と比較するか，負値判定でエラーを検出する。
7. **割込みサービスルーチンは短く保ち**，待ち状態に遷移するサービスコールを呼ばない（`wai_*`, `tslp_tsk` 等は禁止）。タスクへの通知はセマフォ・イベントフラグ・データキューで行う。
8. **`CFG_INT` で割込み要求ラインの属性を設定してから `ATT_ISR` でISRを登録**する順序を守る。
9. **ディスパッチ保留状態（CPUロック・優先度マスク・dis_dsp）からは待ちに入るサービスコールを呼び出さない**。`E_CTX` で必ず失敗する。
10. **タスク優先度は1〜TMAX_TPRI（既定16）の範囲**。値が小さいほど高優先度。割込み優先度は負値で，TMIN_INTPRI（最小値）〜TMAX_INTPRI（=-1）。

## 13. ターゲット定義事項について

本仕様の多くは「ターゲット定義」とされており，移植先のCPUアーキテクチャ・ボード・コンパイラに応じてターゲット依存部の実装が用意される。本書ではターゲット非依存の規定のみを扱う。実装に依存する詳細（最小スタックサイズ・割込み優先度の段階数・割込み番号の付与・FPU属性 `TA_FPU` の有無等）は対象ターゲットの「ターゲット依存部仕様書」を参照する必要がある。

LLMがコード生成時にターゲット依存事項を引用するなら，「ターゲット定義」と明記し，ターゲット仕様書側の値を参照させる旨を断ること。例: スタックサイズの妥当値，割込み番号の具体値，初期化フックの登録方法など。
