# 禁則の詳細と根拠

TOPPERS派生カーネルで**絶対に守る**ルール。違反は上流追従の破綻・安全認証の失効に直結する。

## 禁則① `kernel/` を直接編集しない

### なぜ
`kernel/`・`include/`・`library/` は上流TOPPERSカーネルコアを**ほぼそのまま維持**する
PRISTINE領域。直接編集すると、上流が新バージョンを出すたびに手動マージの差分が増え、
追従が破綻する。派生の価値（上流のバグ修正・改良を取り込み続けられること）が失われる。

### どこに書くか
| やりたいこと | 正しい置き場所 |
|---|---|
| アーキ依存処理（コンテキストスイッチ・割込み・FPU・MPU等） | `arch/<arch>/` |
| ボード/チップ依存処理（メモリマップ・SIO・タイマ・ベクタ） | `target/<name>/` |
| システムサービス（syslog・serial・logtask・banner 等） | `syssvc/` |
| 全く新しい機能 | 新規ファイル（NEW扱い） |

### どうしてもコアに手を入れる場合
1. **作業を止めてユーザーに確認**する。
2. 変更は `#ifdef`（例：機能ガードマクロ）で囲み、**既定OFFで上流と完全に同一動作**を保つ。
3. 乖離台帳（リポジトリが持つ上流乖離の一覧）に「ファイル・変更種別・理由・上流変更時
   リスク・担当レイヤ」を記録。
4. 可能なら上流へ不具合報告し、将来のマージで取り込まれるようにする。

> `#ifdef` ガードは「純追加」を最優先（既存行を `#else` で置換するより、上流変更時の
> 再同期が容易）。`#ifdef/#else` で既存命令を差し替える箇所は最脆弱としてマップに明記する。

## 禁則② カーネル内で動的メモリ確保を使わない

### 禁止
`malloc` / `calloc` / `realloc` / `free` / C++ `new` / `delete`。

### 代替
- **静的配列**（コンパイル時にサイズ確定）
- **固定長メモリプール**：`.cfg` の `CRE_MPF` で静的生成し、`get_mpf`/`pget_mpf`/
  `tget_mpf` で取得、`rel_mpf` で返却。

### なぜ
動的確保はヒープ断片化・確保失敗・実行時間非決定性を生み、ISO 26262 / IEC 61508 が
求める「メモリ使用量の静的な上界保証」を崩す。TOPPERSの安全設計方針はオブジェクトを
**すべてコンフィギュレータがビルド時に静的生成**することで、実行時の確保を排除している。

## その他の必須ルール

- **再帰禁止**：カーネル内で再帰関数を使わない。スタック消費の上界が静的に決まらず、
  スタックオーバーフロー→ハードフォルトの原因になる。ループに展開する。
- **エラーは `ER` 戻り値**：例外（C++）・`longjmp` でのエラー伝播をしない。
  サービスコールの `E_*` 戻り値（負値）は呼び出し側で必ず確認する。
- **FPはタスクコンテキストのみ**：割込みハンドラ・周期/アラームハンドラ内で
  浮動小数点を使わない（FPUコンテキスト退避コストとレイテンシのため）。
- **命名はTOPPERS流**：関数=`act_tsk` 風（小文字+`_`）、型=`ID`/`ER`/`PRI`/`T_CTSK`
  （大文字 typedef）、マクロ/定数=`TMIN_TPRI`/`E_OK`/`TA_NULL`（大文字+`_`）。
  カーネル内部関数は上流の接頭辞・命名に従う。

## 静的解析（該当する場合）

安全クリティカル領域を意識し、MISRA-C:2012 サブセット相当を守る。生成コード・追加
コードがこれに違反しないこと（CIで `cppcheck`/`clang-tidy` が回る構成が多い）。

## PE 別・用途別のベクタ表は 2 のべき乗境界に整列させる

〔分類: 整列・移植/リンカ ｜ プロセッサ: ARMv8-M（ベクタ基底レジスタを持つ M-profile） ｜ 出所: FMP3〕

ベクタ表の基底レジスタを持つアーキでは、cfg が PE 別・用途別に複数のベクタ表を生成する構成のとき、**各表を必要な 2 のべき乗境界に整列**させなければならない。境界が合っていないと基底レジスタが下位ビットを無視し、誤ったアドレスを参照してランダムなベクタが実行される。

- 必要な整列バイト数はエントリ数に依存する（エントリ数×エントリサイズ以上の 2 のべき乗）。
- PE 数やベクタ数が増えるごとに必要整列量が変わるため、cfg のテーブル生成と `.trb` のリンカ指定を一致させる。
- リンカスクリプトで `ALIGN(...)` を指定し忘れると実機・QEMU 双方で誤動作するが、症状がランダムに見えるため見つけにくい。
