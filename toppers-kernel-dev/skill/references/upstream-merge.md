# 上流追従マージの考え方（実装非依存）

TOPPERS派生カーネルの価値は「上流の改良・バグ修正を取り込み続けられること」。
そのために**乖離を明文化した台帳**を持ち、AIが機械的に支援できる形にする。
（台帳ファイルの具体名・場所はリポジトリ固有。本書は役割と手順の原則。）

## 持つべき情報（リポジトリ側に整備されている前提）

| 情報 | 単位 | 役割 |
|---|---|---|
| ベース上流バージョン記録 | — | 現在ベースにしている上流バージョン・取得元・最終マージ日 |
| 乖離台帳 | **ファイル単位** | 上流から意図的に乖離したファイル一覧（変更種別・理由・上流変更時リスク・担当レイヤ・最終確認バージョン） |
| 上流そのまま一覧 | ファイル一覧 | 一切変更していない＝上流版で上書きしてよいファイルの一覧 |
| cfg追従台帳 | cfg層単位 | コンフィギュレータの上流追従（後述の層別） |

乖離台帳に載っていないファイルは原則「上流そのまま」（上流版で上書き可）。

## 変更種別（マージ時の扱い）

| 種別 | 意味 | マージ時 |
|---|---|---|
| 上流そのまま（PRISTINE） | カーネルコア等を無改変 | 上流版で上書き（人手不要） |
| 拡張（EXTENDED） | 上流＋追加 | 上流変更を確認して取り込む |
| 意図的変更（DIVERGED） | 仕様を変えた | 台帳に従い要人間確認 |
| 新規（NEW） | 完全な新規追加 | 上流と衝突なし |

## マージ手順

```
1. 上流リリースノートで変更概要を把握
        ↓
2. 上流diff（現ベース..新版）を取得
        ↓
3. 上流diff × 乖離台帳 を照合し「影響ファイル」を列挙
   ├─ 「上流そのまま」のみ該当 → 上流版で上書き更新（人手不要）
   └─ 乖離台帳に該当           → 「要人間確認」とマークして停止
        ↓
4. cfg関連は層ごとに判定（下記）
        ↓
5. ホストsim → QEMU の順で回帰テスト（機械判定）
        ↓
6. ベース上流バージョン記録を更新してコミット
```

**ステップ3でAIが出す「影響ファイル一覧」が人間の確認スコープを絞る。**
乖離台帳が整備されているほど絞り込み精度が上がる。乖離部に上流変更が及ぶ項目は
停止してユーザー確認する。

## cfg（コンフィギュレータ）の層別マージ

cfgは層ごとにマージ難易度が異なる。区別して管理する：

| 層 | 実体 | マージ性 |
|---|---|---|
| ① 静的API定義 | `kernel/kernel_api.def`（api-table） | **上流と同形式＝テキストマージ可** |
| ② エンジン | コンフィギュレータ本体（言語移植している場合は移植版） | 挙動変更はテキスト差分不可・手動再反映 |
| ③ 生成テンプレート | 生成スクリプト（移植版がある場合は対応版） | 手動再反映 |

判定：
- 上流の変更が**静的APIの追加・変更だけ** → 層①＝`kernel_api.def` の差分マージで済む。
- 上流の**エンジン挙動が変わった** → 層②/③＝対応表と「最終確認バージョン」で更新箇所を特定し再反映。

## コミット規約（一般例）

```
<type>(<scope>): <summary>
type:  feat / fix / docs / test / chore / refactor
scope: target / syssvc / cfg / upstream / ci など
例:    chore(upstream): merge <kernel> <version>
```

ブランチは `upstream`（上流取込）／`main`（常にビルド・テストが通る作業）／
`feat/*`（複数セッション・破壊的変更時）が典型。
